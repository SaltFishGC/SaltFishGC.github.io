## 环境隔离

在学习Nacos的**配置中心**的时候，我们会使用一套**命名空间（namespace）：分组（group）**的隔离空间，其实Nacos的**注册中心**也使用了这么一套隔离空间，这样我们就可以在开发环境，测试环境，生产环境使用一个Nacos注册中心，只需要准备好namespace以及group并指定好服务的空间即可。

### 配置流程

**创建命名空间**：只需要到Nacos的web可视化界面里创建需要的命名空间即可

**获取命名空间ID**：完成创建之后可以看到Nacos分配给新命名空间的ID，记录下来

**服务中指明注册中心的命名空间**：在yml配置中的discovery项下加入namespace项，并指定**命名空间的ID**

```YAML
spring:
  application:
    name: item-service # 服务名称
  profiles:
    active: dev
  cloud:
    nacos:
      server-addr: 192.168.150.101 # nacos地址
      discovery: # 服务发现配置
        namespace: 8c468c63-b650-48da-a632-311c75e6d235 # 设置namespace，必须用id
      # 。。。略
```

> 必须指定**命名空间ID**！而不是命名空间的name！
> 一个服务实例的**注册中心**只能指定一个命名空间！
>
> 但是配置中心和注册中心的可以分别是两个：
>
> ```yml
> # application.yml 或 bootstrap.yml
> spring:
>   cloud:
>     nacos:
>       # ===== 注册中心配置 =====
>       discovery:
>         server-addr: 127.0.0.1:8848
>         namespace: dev-ns-id          # 服务注册到 dev 命名空间
>         service: user-service
>         group: DEFAULT_GROUP
>         
>       # ===== 配置中心配置 =====
>       config:
>         server-addr: 127.0.0.1:8848
>         namespace: common-ns-id       # 配置从 common 命名空间读取
>         file-extension: yaml
>         share-configs:
>           - data-id: shared-common.yaml
>             group: DEFAULT_GROUP
>             refresh: true
> ```



## 服务分级模型

```
┌─────────────────────────────────────────────────────────────┐
│                    Namespace（命名空间）                      │
│                    环境隔离级别（最高级）                      │
│  ┌───────────────────────────────────────────────────────┐   │
│  │                  Group（分组）                         │   │
│  │                  业务分组级别                          │   │
│  │  ┌─────────────────────────────────────────────────┐  │   │
│  │  │              Service（服务）                     │  │   │
│  │  │              服务名称级别                        │  │   │
│  │  │  ┌───────────────────────────────────────────┐  │  │   │
│  │  │  │          Cluster（集群）                   │  │  │   │
│  │  │  │          机房/区域级别                     │  │  │   │
│  │  │  │  ┌─────────────┐  ┌─────────────┐        │  │  │   │
│  │  │  │  │  Instance   │  │  Instance   │        │  │  │   │
│  │  │  │  │  实例1       │  │  实例2      │        │  │  │   │
│  │  │  │  └─────────────┘  └─────────────┘        │  │  │   │
│  │  │  └───────────────────────────────────────────┘  │  │   │
│  │  └─────────────────────────────────────────────────┘  │   │
│  └───────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

Nacos 注册中心采用 **四级分层架构** 来组织和管理微服务实例，从大到小依次为：

```
Namespace（命名空间） → Group（分组） → Service（服务） → Cluster/Instance（集群/实例）
```

内部维护的Map源码：

```
┌─────────────────────────────────────────────────────────────────┐
│                      Nacos Server 注册表                         │
├─────────────────────────────────────────────────────────────────┤
│  Map<Namespace,                                                 │
│    Map<Group,                                                   │
│      Map<ServiceName,                                           │
│        Service {                                                │
│          Map<ClusterName, Cluster {                             │
│            List<Instance>  ← 实例列表                            │
│          }>                                                     │
│        }>                                                       │
│    >                                                            │
│  >                                                              │
└─────────────────────────────────────────────────────────────────┘
```

| 类名       | 作用                                     |
| :--------- | :--------------------------------------- |
| `Service`  | 服务元数据，包含服务名、分组、保护阈值等 |
| `Cluster`  | 集群信息，包含集群名、健康检查配置等     |
| `Instance` | 实例信息，包含IP、端口、权重、健康状态等 |

> 微服务实例在不同隔离环境下的相互调用：
>
> | 场景                        | 是否可以直接调用     | 说明                           |
> | :-------------------------- | :------------------- | :----------------------------- |
> | **同一命名空间 + 同一分组** | ✅ 可以               | 默认行为，无需额外配置         |
> | **同一命名空间 + 不同分组** | ⚠️ 可以，但需指定分组 | 需要在调用时明确指定目标分组   |
> | **不同命名空间 + 任意分组** | ❌ 不可以             | 命名空间是强隔离，默认无法发现 |

### 服务集群

**误解：**Service 下面的所有实例 = 一个集群**❌**

**实际结构：**

```
Service（服务）
├─ Cluster A（集群A）
│   ├─ 实例1
│   ├─ 实例2
│   └─ 实例3
├─ Cluster B（集群B）
│   ├─ 实例4
│   ├─ 实例5
│   └─ 实例6
└─ Cluster C（集群C）
    ├─ 实例7
    └─ 实例8
```

为什么需要再嵌套一次集群？

 **核心目的：就近调用 + 故障隔离**

```
┌─────────────────────────────────────────────────────────────────┐
│                        user-service                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────┐      ┌─────────────────────┐         │
│   │  Cluster: HANGZHOU  │      │  Cluster: SHANGHAI  │         │
│   │  (杭州机房)          │      │  (上海机房)          │         │
│   ├─────────────────────┤      ├─────────────────────┤         │
│   │  192.168.1.10:8080  │      │  192.168.2.10:8080  │         │
│   │  192.168.1.11:8080  │      │  192.168.2.11:8080  │         │
│   │  192.168.1.12:8080  │      │  192.168.2.12:8080  │         │
│   └─────────────────────┘      └─────────────────────┘         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**调用规则：**

- 杭州的消费者 → 优先调用杭州集群
- 上海的消费者 → 优先调用上海集群
- 某机房故障 → 自动切换到另一机房

如果不分集群：

```
┌─────────────────────────────────────────────────────────────────┐
│                        user-service                             │
│                    (没有 Cluster 分层)                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   192.168.1.10:8080 (杭州)    192.168.2.10:8080 (上海)           │
│   192.168.1.11:8080 (杭州)    192.168.2.11:8080 (上海)           │
│   192.168.1.12:8080 (杭州)    192.168.2.12:8080 (上海)           │
│                                                                 │
│   ❌ 问题：杭州消费者可能调用到上海实例，增加网络延迟      	         │
│   ❌ 问题：杭州机房故障，无法快速隔离，影响上海机房          		      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> 方便LoadBalancer负载均衡策略，**对实例进行更细粒度的分组管理**
