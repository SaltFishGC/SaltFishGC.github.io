## CAP定理

CAP 定理（也称为 Brewer 定理）是分布式系统设计中的一个核心理论，由计算机科学家 Eric Brewer 在 2000 年提出，并在 2002 年由 Seth Gilbert 和 Nancy Lynch 从理论上严格证明。CAP 定理指出，在一个**分布式数据存储系统**中，**一致性（Consistency）、可用性（Availability）和分区容错性（Partition tolerance）** 这三个属性**无法同时满足**，最多**只能同时满足其中的两个**。

- **C – Consistency（一致性）**（高安全）（XA）
  指的是：所有节点在同一时间看到的数据是一致的。即，对系统的任何读操作都能返回最近一次成功写入的结果。这通常指“强一致性”。
- **A – Availability（可用性）**（高性能）（AT）
  指的是：系统始终能够响应客户端的请求（读或写），即使某些节点发生故障。注意：响应不一定是“成功”，但必须在有限时间内返回结果（不能无限期挂起或超时）。
- **P – Partition tolerance（分区容错性）**
  指的是：当网络发生分区（即部分节点之间通信中断）时，系统仍能继续运行。由于网络不可靠是现实常态（如断网、延迟等），**实际的分布式系统必须具备 P**。

为什么说**只能三选二**：

- 网络分区（P）在真实世界中不可避免，因此**几乎所有实用的分布式系统都必须选择 P**。
- 一旦选择了 P，就只能在 C 和 A 之间做权衡：
  - 如果选择 **CP**：在网络分区发生时，为了保证一致性，系统可能拒绝部分请求（牺牲可用性）。例如：ZooKeeper、etcd。
  - 如果选择 **AP**：在网络分区发生时，系统继续接受读写请求，但可能导致不同节点看到的数据不一致（牺牲一致性）。例如：Cassandra、DynamoDB。

> 注意：CAP 中的“放弃”并不是完全不要，而是在**网络分区发生期间**暂时牺牲某一项。

| 系统            | CAP 选择               | 说明                        |
| :-------------- | :--------------------- | :-------------------------- |
| ZooKeeper       | CP                     | 强一致性，分区时可能不可用  |
| etcd            | CP                     | Kubernetes 使用，强调一致性 |
| Cassandra       | AP                     | 高可用，支持最终一致性      |
| DynamoDB        | AP（可配置为 CA-like） | 默认高可用，提供可调一致性  |
| MongoDB（早期） | CP（副本集）           | 后来支持可调一致性          |

- CAP 定理的核心启示

  ：在设计分布式系统时，必须根据业务需求明确优先级。

  - 金融交易系统 → 更看重一致性（CP）
  - 社交媒体、日志系统 → 更看重可用性（AP）

- **现代系统趋势**：通过“可调一致性”、“多活架构”、“异步复制”等技术，在 C 和 A 之间实现灵活平衡。



## Base理论

**BASE 理论** 是对 **CAP 定理中 AP（可用性 + 分区容错性）系统** 的一种设计哲学和实践指导，特别适用于那些**牺牲强一致性、追求高可用和可扩展性**的分布式系统（如大型互联网应用）。

BASE 是三个英文词的缩写：

- **B – Basically Available（基本可用）**
- **A – Soft state（软状态）**
- **S – Eventually consistent（最终一致性）**

它由 eBay 的架构师 Dan Pritchett 在 2008 年提出，作为对传统 ACID（原子性、一致性、隔离性、持久性）事务模型在分布式场景下的一种“柔性”替代方案。

> 💡 简单说：**BASE = 放弃强一致，换取高可用和伸缩性**。

### **1.Basically Available（基本可用**

系统在出现故障（如部分节点宕机、网络延迟）时，**仍能提供核心功能**，但可能：

- 响应变慢（如增加几秒加载时间）
- 功能降级（如只读模式、简化页面）
- 部分数据暂时不可用

> ✅ 举例：电商大促时，下单成功但“订单详情”稍后才能查看——系统优先保证下单可用。

### **2.Soft state（软状态**

系统状态**可以有一段时间是“不一致”或“中间态”**，不需要时刻保持同步。

- 数据副本之间允许存在**临时差异**
- 状态可能依赖于**时间、事件或外部输入**而变化

> ✅ 举例：用户修改头像后，好友看到的还是旧头像——这是“软状态”，后续会更新。

### **3.Eventually consistent（最终一致性**

这是 BASE 的核心。

- 系统**不要求实时一致**，但保证：**如果没有新的更新操作，经过一段时间后，所有节点的数据会趋于一致**。
- “一段时间”取决于网络延迟、同步机制等，可能是毫秒级，也可能是几分钟。

> ✅ 举例：你在微信发一条朋友圈，你的朋友可能 1 秒内看到，也可能因缓存延迟 5 秒后才看到——但最终大家看到的内容是一样的。

| 特性         | ACID（传统数据库）             | BASE（分布式系统）             |
| :----------- | :----------------------------- | :----------------------------- |
| 一致性       | 强一致性（立即一致）           | 最终一致性（延迟一致）         |
| 可用性       | 可能因锁/事务阻塞              | 高可用，容忍部分失败           |
| 适用场景     | 银行转账、库存扣减等强一致场景 | 社交媒体、推荐系统、日志分析等 |
| 性能与扩展性 | 扩展性较差（尤其写操作）       | 易水平扩展，适合海量数据       |

> 🔁 实际中很多系统采用 **混合策略**：核心交易用 ACID，非核心功能用 BASE。



### 应用场景

1. **社交网络**
   - 发帖、点赞、评论不需要立即全局一致
   - 用户容忍短暂延迟
2. **电商平台**
   - 商品浏览、推荐列表可用最终一致性
   - 但**下单、支付**仍需强一致（用 ACID 或分布式事务）
3. **内容分发网络（CDN）**
   - 缓存更新采用最终一致性
   - 提升访问速度和可用性
4. **NoSQL 数据库**
   - 如 Cassandra、DynamoDB、Riak 默认采用 BASE 模型

**BASE 是分布式系统在 CAP 定理约束下，选择 AP 路线后的自然延伸**。
它用“柔性”的一致性换来了系统的**高可用、高性能和可扩展性**，非常适合现代互联网大规模应用场景。



## AT模式的脏写问题

首先回顾一下AT的2PC流程：

1. **第一阶段（Prepare）**：
   - 执行业务 SQL（如 `UPDATE account SET balance = balance - 100 WHERE id = 1`）
   - Seata 拦截 SQL，**解析并保存修改前的镜像（before image）和修改后的镜像（after image）到 undo_log 表**
   - 本地事务提交（此时数据已真实写入 DB）
2. **第二阶段（Commit / Rollback）**：
   - 如果全局事务成功 → 删除 undo_log（异步）
   - 如果失败 → 用 before image **反向生成 SQL 回滚**（如 `UPDATE account SET balance = 1000 WHERE id = 1`）

> ⚠️ 注意：**第一阶段就提交了本地事务**！这是 AT 模式高性能的原因，但也埋下了“脏写”的隐患。

在 AT 模式中，**脏写指的是：在全局事务未最终提交前，其他并发事务读取或修改了该事务已提交但可能被回滚的数据**。

由于 AT 模式 **第一阶段就释放了本地数据库行锁**（因为本地事务已提交），其他事务可以立即访问这些“中间状态”数据。

 **场景 1：脏读（Dirty Read）**

- 事务 T1 修改了数据并本地提交（但全局未决）
- 事务 T2 读到了 T1 写入的值
- 后续 T1 全局回滚 → T2 读到的是“从未存在过的数据”

> 虽然脏读本身是读问题，但在某些业务中会引发错误决策。

 **场景 2：脏写（Dirty Write / Write Skew）更危险！**

- 事务 T1 修改了数据 A（本地已提交，全局未决）
- 事务 T2 **基于 T1 的中间状态** 修改了数据 A（或相关数据）
- T1 最终回折 → T2 的修改建立在“无效数据”之上，导致**逻辑错误或数据不一致**

发生脏写的根本原因：
✅ **本地事务提前提交** + ❌ **缺乏对“中间状态数据”的并发控制**

- 数据库行锁在本地事务提交后就释放了
- 其他事务（尤其是非 Seata 管理的本地事务）无法感知“这个数据属于一个未决的全局事务”

> 🔒 Seata 虽然有 **全局锁（Global Lock）**，但**只在 Seata 参与的微服务之间生效**！
> 对于**非 Seata 事务（如 DBA 直连、定时任务、外部系统）完全无效**。

### Seata给出的解决方案-全局锁

- 所有对共享数据的访问**都必须经过 Seata 代理的数据源**
- Seata 会在第一阶段获取**全局行锁**（记录在 TC 中）
- 其他 Seata 事务在尝试修改同一行时会**阻塞或失败**，直到全局事务结束

> ✅ 有效防止 **Seata 事务之间**的脏写
> ❌ 无法防止**非 Seata 事务**的干扰



## TCC模式

**TCC** 是三个英文单词的缩写：

| 阶段  | 英文    | 中文 | 作用          |
| :---- | :------ | :--- | :------------ |
| **T** | Try     | 尝试 | 资源预留/检查 |
| **C** | Confirm | 确认 | 真正执行业务  |
| **C** | Cancel  | 取消 | 释放预留资源  |

> 💡 **核心思想**：将业务逻辑拆分成两个阶段，通过**补偿机制**实现分布式事务的最终一致性。

TCC模式与AT模式非常相似，每阶段都是独立事务，不同的是TCC通过人工编码来实现数据恢复。TCC 模式由 **业务代码主动实现** 三个接口，属于**应用层的事务方案**，与 AT 模式的"无侵入"形成鲜明对比。

###  第一阶段：Try（尝试）

**目的**：完成业务检查 + 资源预留

**要做的事**：

- ✅ 检查业务参数合法性
- ✅ 检查业务规则（如余额是否充足）
- ✅ **预留资源**（如冻结资金、预占库存）
- ✅ 记录预留日志（用于后续 Confirm/Cancel）

**特点**：

- Try 操作必须是**幂等**的
- Try 失败则直接返回，不进入后续阶段
- Try 成功后，资源处于"预留状态"，**对其他事务不可用**

###  第二阶段：Confirm（确认）

**目的**：真正执行业务，提交资源

**要做的事**：

- ✅ 使用 Try 阶段预留的资源
- ✅ 执行真实的业务操作
- ✅ 释放预留状态

**特点**：

- Confirm **必须成功**（设计时要保证）
- Confirm 操作必须是**幂等**的
- Confirm 失败会**持续重试**，直到成功

###  第二阶段：Cancel（取消）

**目的**：释放 Try 阶段预留的资源

**要做的事**：

- ✅ 释放冻结的资金
- ✅ 归还预占的库存
- ✅ 清理预留日志

**特点**：

- Cancel 操作必须是**幂等**的
- Cancel 失败会**持续重试**，直到成功
- Cancel 必须能处理"Try 成功但 Confirm 未执行"的情况

### 大致流程图：

```
┌─────────────────────────────────────────────────────────────────┐
│                        全局事务开始                               │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                     第一阶段：Try（所有分支）                       │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐          │
│  │  服务 A Try  │    │  服务 B Try  │    │  服务 C Try  │         │
│  │  (资源预留)  │    │  (资源预留)  │    │  (资源预留)  │           │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘          │
│         │                  │                  │                 │

```

### 和AT的区别

| 特性                  | AT 模式                  | TCC 模式                 |
| :-------------------- | :----------------------- | :----------------------- |
| **侵入性**            | 无侵入（只需注解）       | 高侵入（需实现三个接口） |
| **事务粒度**          | 数据库行级               | 业务方法级               |
| **隔离性**            | 第一阶段提交，有脏写风险 | 资源预留，隔离性好       |
| **性能**              | 高（本地事务直接提交）   | 中（需额外预留逻辑）     |
| **适用场景**          | 简单 CRUD 场景           | 复杂业务、跨系统调用     |
| **回滚方式**          | 自动 undo log 回滚       | 手动 Cancel 补偿         |
| **非 Seata 事务兼容** | ❌ 不兼容                 | ✅ 可兼容（业务控制）     |
| **开发成本**          | 低                       | 高                       |

> **TCC 模式中的"资源预留"必须由业务方自己实现！**



## 最大努力通知

**最大努力通知** 是一种**异步最终一致性**的分布式事务解决方案。

> 💡 **核心思想**：事务发起方**尽最大努力**通知相关服务执行操作，但不保证 100% 成功。如果通知失败，会通过**重试机制**反复尝试，直到成功或达到最大重试次数。

**通俗理解**

```
┌─────────────┐         ┌─────────────┐
│  事务发起方   │  ────→  │  事务接收方  │
│  (订单服务)  │  通知    │  (积分服务)  │
└─────────────┘         └─────────────┘
       │                       │
       │  "我尽量通知你，        │
       │   但不保证一定成功"      │
       │                       │
       ▼                       ▼
  多次重试                   最终执行
```

> 消息通知可以使用MQ，或者直接对外开放接口来实现

| 场景                 | 问题                   | 最大努力通知的作用   |
| :------------------- | :--------------------- | :------------------- |
| 订单支付成功后送积分 | 积分服务可能暂时不可用 | 重试直到积分服务恢复 |
| 注册成功后发短信     | 短信网关可能超时       | 多次尝试发送         |
| 下单成功后通知物流   | 物流系统可能宕机       | 保证最终通知到       |
| 支付成功后更新统计   | 分析系统可能延迟       | 异步最终一致即可     |

> 🔑 **关键点**：这些场景**不需要强一致性**，但需要**保证最终能通知到**。

这样我们就可以不去使用其他的seata之类的依赖或者是分布式模型了，流程：

```
┌─────────────────────────────────────────────────────────────────┐
│                      最大努力通知流程                            │
└─────────────────────────────────────────────────────────────────┘

  步骤 1: 本地事务提交
  ┌─────────────────────────────────────────┐
  │  1. 执行本地业务（如创建订单、扣款）     │
  │  2. 本地事务提交                         │
  │  3. 记录通知日志（待通知状态）           │
  └─────────────────────────────────────────┘
                          │
                          ▼
  步骤 2: 第一次通知
  ┌─────────────────────────────────────────┐
  │  1. 同步/异步调用接收方接口              │
  │  2. 成功 → 更新通知日志为"已通知"        │
  │  3. 失败 → 记录失败原因，等待重试        │
  └─────────────────────────────────────────┘
                          │
                          ▼
  步骤 3: 重试机制
  ┌─────────────────────────────────────────┐
  │  1. 定时任务扫描"待通知"记录             │
  │  2. 按策略重试（如 1min, 5min, 30min...）│
  │  3. 达到最大次数 → 标记"通知失败"        │
  │  4. 人工介入处理                         │
  └─────────────────────────────────────────┘
                          │
                          ▼
  步骤 4: 接收方幂等处理
  ┌─────────────────────────────────────────┐
  │  1. 接收方检查是否已处理过               │
  │  2. 已处理 → 直接返回成功                │
  │  3. 未处理 → 执行业务逻辑                │
  └─────────────────────────────────────────┘
```



## 总结

避免分布式问题的最优策略就是不用分布式，在项目的性能需求没有到达单机处理不了的地步之前，尽量避免使用分布式！
