## 线程隔离

**线程隔离（Thread Isolation）** 是一种服务保护机制，通过为不同的服务调用分配**独立的线程池**，防止某个服务的故障（如响应慢、超时、异常）耗尽所有线程资源，导致整个系统雪崩。

线程隔离有两种方式实现：

- **线程池** **隔离**：给每个服务调用业务分配一个线程池，利用线程池本身实现隔离效果
- **信号量** **隔离**：不创建线程池，而是计数器模式，记录业务使用的线程数量，达到信号量上限时，禁止新的请求

| 维度         | 线程隔离              | 信号量隔离                      |
| :----------- | :-------------------- | :------------------------------ |
| **实现方式** | 独立线程池            | 计数器（Semaphore）             |
| **资源消耗** | 高（线程上下文切换）  | 低（仅计数）                    |
| **隔离效果** | 强（完全隔离）        | 中（共享线程）                  |
| **异步支持** | 支持                  | 不支持                          |
| **超时控制** | 支持                  | 支持                            |
| **适用场景** | 慢调用、可能阻塞      | 快速调用、高并发                |
| **代表框架** | Hystrix（线程池模式） | Sentinel、Hystrix（信号量模式） |



## 滑动窗口算法

### 固定窗口算法

**固定窗口（Fixed Window）** 是最简单的限流算法，将时间划分为固定长度的时间段（如每秒、每分钟），在每个时间段内维护一个计数器，当请求数超过阈值时拒绝后续请求。

- 将时间划分为多个窗口，窗口时间跨度称为`Interval`，本例中为1000ms；
- 每个窗口维护1个计数器，每有1次请求就将计数器`+1`。限流就是设置计数器阈值，本例为3，图中红线标记
- 如果计数器超过了限流阈值，则超出阈值的请求都被丢弃。

**时间划分示例**

假设窗口大小为 1 秒，限流阈值为 3：

```
时间轴：|----------|----------|----------|----------|
       [0s, 1s)   [1s, 2s)   [2s, 3s)   [3s, 4s)
		  窗口1      窗口2       窗口3      窗口4

每个窗口最多允许 3 个请求
```

**请求处理流程**

1. **确定当前窗口**：根据当前时间计算所属窗口
2. **获取窗口计数器**：找到对应窗口的计数值
3. **判断是否超限**：如果计数 < 阈值，允许请求并计数+1
4. **窗口切换**：进入新窗口时，计数器重置为 0



### 滑动窗口算法

滑动窗口计数（Sliding Window Counting）是固定窗口算法的改进版本，通过**连续统计最近一段时间内的请求总数**来解决固定窗口的临界突刺问题。

**与固定窗口的本质区别**

- **固定窗口**：将时间切成离散的块，每个块独立计数
- **滑动窗口计数**：维护一个连续的时间段，统计这个时间段内的总请求数

**工作原理**

```
假设：窗口大小 1 秒，限流阈值 5

当前时间：1.2 秒
统计范围：[0.2, 1.2] 这个连续的 1 秒区间

而不是固定窗口的 [1.0, 2.0) 这种离散区间
```

这样就避免了在窗口边界处出现流量翻倍的问题。



## 漏桶算法

漏桶算法（Leaky Bucket Algorithm）是一种经典的**流量整形**（Traffic Shaping）和**速率限制**（Rate Limiting）算法，常用于**服务保护**场景中，以防止系统因突发流量而过载。其核心思想是将请求“装入”一个固定容量的“桶”中，并以恒定速率“漏出”处理，从而平滑突发流量。

想象有一个底部有小孔的水桶：

- **请求到达**：相当于往桶里加水。
- **请求处理**：相当于水从桶底的小孔以**固定速率**流出。
- **桶的容量有限**：如果桶满了，新来的水（请求）就会溢出，即被**丢弃或拒绝**。

因此，无论输入流量多么突发，输出流量始终是**匀速且可控的**。

**请求流程**：

1. 当一个请求到来时：
   - 如果桶未满，则将请求放入桶中（排队等待处理）。
   - 如果桶已满，则拒绝该请求（返回错误或限流提示）。
2. 系统以固定速率从桶中取出请求进行处理（无论是否有新请求到达）。

> 注意：漏桶算法通常使用一个**队列**来实现“桶”，并由一个定时器或后台线程按固定速率消费队列中的请求。

| 特性             | 说明                                                 |
| :--------------- | :--------------------------------------------------- |
| **平滑输出**     | 输出速率恒定，有效削峰填谷                           |
| **抗突发能力弱** | 即使系统空闲，也不能临时提高处理速率（与令牌桶不同） |
| **可能增加延迟** | 请求可能在桶中排队等待处理                           |
| **严格限流**     | 能保证后端服务不会超负荷                             |

漏桶算法通过**固定速率处理请求**和**有限缓冲区**，为后端服务提供稳定、可预测的负载，是服务保护体系中的重要组件。虽然它牺牲了对突发流量的容忍度，但在需要严格控制流量节奏的场景中非常有效。



## 令牌桶算法

令牌桶算法（Token Bucket Algorithm）是一种广泛应用于**限流**（Rate Limiting）和**流量整形**（Traffic Shaping）的经典算法，特别适合需要**允许一定程度突发流量**的服务保护场景。与漏桶算法不同，令牌桶更灵活，既能限制平均速率，又能在系统空闲时“积攒”处理能力以应对短时高峰。

想象一个以固定速率不断生成“令牌”的桶：

- **令牌生成**：系统以恒定速率（如每秒10个）向桶中添加令牌。
- **桶有容量上限**：桶最多只能存放 `capacity` 个令牌，超过则丢弃新生成的令牌。
- 请求处理：每个请求要被处理，必须从桶中消耗一个令牌。
  - 如果桶中有令牌 → 请求被允许；
  - 如果桶中无令牌 → 请求被拒绝或排队等待。

> 关键思想：**令牌代表处理许可**，有令牌才能通行。

**请求流程：**

1. 系统后台持续以固定速率往桶里加令牌（不超过容量）。
2. 当请求到达时：
   - 检查桶中是否有令牌；
   - 有 → 消耗一个令牌，放行请求；
   - 无 → 拒绝请求（或进入等待队列，视实现而定）。

> 实际实现中通常**不使用后台线程**，而是通过**时间差计算**动态推算当前令牌数，提高效率。

| 对比项           | 漏桶算法                      | 令牌桶算法                          |
| :--------------- | :---------------------------- | :---------------------------------- |
| 流出方式         | 恒定速率处理请求              | 允许突发（只要有令牌）              |
| 是否允许突发流量 | ❌ 否                          | ✅ 是                                |
| 实现视角         | 请求排队等待处理              | 系统发放“通行证”（令牌）            |
| 适用场景         | 需要严格平滑流量（如网络QoS） | 需要兼顾限流与突发弹性（如API网关） |

简单记忆：

- **令牌桶**：**“有令牌就能走”** → 更灵活；
- **漏桶**：**“按节奏慢慢走”** → 更平稳。
