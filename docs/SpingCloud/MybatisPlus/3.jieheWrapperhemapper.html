结合Wrapper和mapper自定义sql

SaltFishGC2026-1-29

在 MyBatis-Plus（MP）中，结合 Wrapper（条件构造器）和 Mapper 自定义 SQL 是非常常见的需求： 你希望 复用自定义的复杂 SQL（如多表 JOIN、子查询、聚合等），同时又能 动态传入 WHERE 条件 —— 而这些条件最好由 MP 的 Wrapper 来构建，避免手动拼接。

好消息是：MyBatis-Plus 完全支持这种混合模式！

✅ 核心思路

在自定义 SQL 中使用 <where> + ${ew.sqlSegment}（或 ${ew.customSqlSegment}）， 然后在 Mapper 接口中方法参数加上 @Param(Constants.WRAPPER) Wrapper wrapper。

这样，MP 会自动将 Wrapper 构建的条件拼接到你的 SQL 中。

一、具体步骤（以 LambdaQueryWrapper 为例）

1. Mapper 接口定义

public interface UserMapper extends BaseMapper<User> {

    // 自定义查询方法，接收 Wrapper 作为条件
    List<UserVO> selectUserWithDept(@Param(Constants.WRAPPER) Wrapper<User> wrapper);
}

🔸 必须使用 @Param(Constants.WRAPPER) 注解，且参数名固定为 wrapper（或任意名，但注解值必须是 Constants.WRAPPER）。 🔸 返回类型可以是 VO、Map 或实体，不强制是 Entity。

2. XML 中编写自定义 SQL

<select id="selectUserWithDept" resultType="com.example.vo.UserVO">
    SELECT 
        u.id, u.name, u.email,
        d.dept_name AS deptName
    FROM user u
    LEFT JOIN department d ON u.dept_id = d.id
    <where> 
        ${ew.sqlSegment}
    </where>
</select>

🔸 ${ew.sqlSegment} 会自动替换为 Wrapper 生成的 WHERE 条件（不含 WHERE 关键字）。 🔸 使用 <where> 标签可自动处理 AND/OR 开头的问题。

3. 调用时传入 Wrapper

@Autowired
private UserMapper userMapper;

public void test() {
    LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<User>()
        .likeRight(User::getName, "张")
        .ge(User::getAge, 18)
        .eq(User::getStatus, 1);

    List<UserVO> list = userMapper.selectUserWithDept(wrapper);
}

生成的 SQL 类似：

SELECT u.id, u.name, ..., d.dept_name
FROM user u
LEFT JOIN department d ON u.dept_id = d.id
WHERE u.name LIKE '张%' AND u.age >= 18 AND u.status = 1

二、注意事项 & 高级技巧

✅ 1. 字段名要对应数据库列（不是 Java 属性）

Wrapper 中的字段最终会转成 数据库列名。 如果你的 Entity 用了 @TableField("user_name")，那么：

wrapper.eq(User::getUserName, "xxx") 
// 生成的是 user_name = 'xxx'

所以你的自定义 SQL 中也要用 数据库列名（如 u.user_name）。

✅ 2. 如果只想用条件部分（不要 ORDER BY / GROUP BY）

默认 ew.sqlSegment 包含 WHERE ... ORDER BY ... GROUP BY ...。

如果你只想要 WHERE 条件部分（比如你自己控制排序），用：

<where>
    ${ew.customSqlSegment}
</where>

customSqlSegment 只包含 WHERE 条件，不包含 ORDER/GROUP/LIMIT。

✅ 3. 支持 Update 场景吗？

支持！ 例如自定义更新：

// Mapper 接口
int updateUserStatus(@Param(Constants.WRAPPER) Wrapper<User> wrapper);
<update id="updateUserStatus">
    UPDATE user
    SET status = 0
    <where>
        ${ew.sqlSegment}
    </where>
</update>

调用：

wrapper.eq(User::getAge, 16);
userMapper.updateUserStatus(wrapper); // 所有 age=16 的用户 status 设为 0

❌ 常见错误

错误原因
Property 'ew' not found忘记加 @Param(Constants.WRAPPER)
字段名对不上Entity 的 Java 属性 vs 数据库列名不一致，且没用 @TableField
SQL 注入警告用了 ${}，但 MP 对 sqlSegment 做了安全过滤（基于字段白名单），一般安全

💡 MP 的 sqlSegment 是安全的，因为它只允许 Entity 中定义的字段，不会直接拼接外部字符串。

三、完整示例（含 VO）

Entity

public class User {
    private Long id;
    @TableField("user_name")
    private String userName;
    private Integer age;
    // getter/setter
}

VO

public class UserVO {
    private Long id;
    private String userName;
    private String deptName;
    // getter/setter
}

Mapper

List<UserVO> selectCustom(@Param(Constants.WRAPPER) Wrapper<User> wrapper);

XML

<select id="selectCustom" resultType="com.example.vo.UserVO">
    SELECT u.id, u.user_name, d.dept_name
    FROM user u
    LEFT JOIN dept d ON u.dept_id = d.id
    <where>
        ${ew.customSqlSegment}
    </where>
    ORDER BY u.id DESC
</select>

调用

var wrapper = new LambdaQueryWrapper<User>()
    .like(User::getUserName, "张")
    .lt(User::getAge, 30);

List<UserVO> list = userMapper.selectCustom(wrapper);

✅ 总结

要点说明
接口参数用 @Param(Constants.WRAPPER) Wrapper<T> wrapper
XML 占位符用 ${ew.sqlSegment}（含 ORDER/GROUP）或 ${ew.customSqlSegment}（仅 WHERE）
字段安全只能使用 Entity 中定义的字段，防注入
适用场景复杂查询（JOIN/子查询）+ 动态条件

这种方式完美融合了 MyBatis 的灵活性 和 MyBatis-Plus 的便捷性，是企业级开发的推荐实践。
