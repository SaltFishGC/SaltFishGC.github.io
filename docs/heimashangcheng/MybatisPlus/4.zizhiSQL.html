自制SQL

SaltFishGC2026-1-29

面对复杂业务（多表）的时候我们难免还是要自己去写sql，前面讲了mapper结合条件构造器Wrapper的自制sql，现在我们来好好说说注解和mapper自制sql的详细使用：

在 MyBatis-Plus（MP）中，SQL 的编写方式主要有两种：

  1. 注解式 SQL（Annotation-based SQL）
  2. XML 映射文件 SQL（Mapper XML SQL）

两者可单独使用，也可混合使用。下面从写法、适用场景、优缺点、详细示例等方面详细说明。

一、注解式 SQL（Annotation-based SQL）

✅ 1. 什么是注解式 SQL？

直接在 Mapper 接口的方法上使用 MyBatis 提供的注解（如 @Select, @Insert 等）来写 SQL，无需 XML 文件。

✅ 2. 常用注解

注解用途
@Select("SQL")查询
@Insert("SQL")插入
@Update("SQL")更新
@Delete("SQL")删除
@Param("name")参数命名（必须！）

⚠️ MP 本身不提供新注解，而是复用 MyBatis 原生注解。

✅ 3. 基本写法示例

public interface UserMapper extends BaseMapper<User> {

    // 简单查询
    @Select("SELECT * FROM user WHERE id = #{id}")
    User selectByIdCustom(Long id);

    // 带条件的动态查询（需用 <script>）
    @Select({
        "<script>",
        "SELECT * FROM user",
        "WHERE status = #{status}",
        "<if test='name != null and name != \"\"'>",
        "  AND name LIKE CONCAT(#{name}, '%')",
        "</if>",
        "</script>"
    })
    List<User> selectByCondition(@Param("status") Integer status, @Param("name") String name);

    // 插入（返回主键）
    @Insert("INSERT INTO user(name, age) VALUES(#{name}, #{age})")
    @Options(useGeneratedKeys = true, keyProperty = "id", keyColumn = "id")
    int insertUser(User user);
}

🔸 注意：

  • 必须用 @Param 标注参数名（除非只有一个参数且是基本类型）。
  • 动态 SQL 需包裹在 <script>...</script> 中。
  • 主键回填需用 @Options。

✅ 4. 优点与缺点

优点缺点
无需 XML，代码集中SQL 可读性差（尤其复杂 SQL）
适合简单、短小的 SQL动态 SQL 写起来繁琐（字符串拼接）
启动快（无 XML 解析）不支持 resultMap（复杂映射困难）
适合快速原型开发重构时字段名硬编码易出错

📌 适用场景：简单单表操作、临时调试、轻量级项目。

二、Mapper XML SQL（推荐主流方式）

✅ 1. 什么是 Mapper XML？

将 SQL 写在独立的 .xml 文件中，通过 namespace 与 Mapper 接口绑定。

✅ 2. 基本结构

（1）Mapper 接口

public interface UserMapper extends BaseMapper<User> {
    
    List<UserVO> selectUserWithDept(@Param("deptId") Long deptId);
    
    void batchUpdateStatus(@Param("ids") List<Long> ids, @Param("status") Integer status);
    
}

（2）XML 文件（resources/mapper/UserMapper.xml）

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">

<mapper namespace="com.example.mapper.UserMapper">

    <!-- 查询：多表 JOIN -->
    <select id="selectUserWithDept" resultType="com.example.vo.UserVO">
        SELECT 
            u.id, u.name, d.dept_name AS deptName
        FROM user u
        LEFT JOIN department d ON u.dept_id = d.id
        WHERE d.id = #{deptId}
    </select>

    <!-- 批量更新 -->
    <update id="batchUpdateStatus">
        UPDATE user
        SET status = #{status}
        WHERE id IN
        <foreach collection="ids" item="id" open="(" separator="," close=")">
            #{id}
        </foreach>
    </update>

</mapper>

（3）配置扫描路径（application.yml）（xml文件路径）

mybatis-plus:
  mapper-locations: classpath*:mapper/**/*.xml
  type-aliases-package: com.example.entity

但很多人会混淆 接口路径 和 XML 文件路径。

  • 接口路径：由 @MapperScan 指定。
  • XML 文件路径：由 MyBatis 的 mapperLocations 配置决定。
项目类型配置项作用
Spring Boot@MapperScan("xxx")扫描 Mapper 接口
Spring Bootmybatis-plus.mapper-locations扫描 Mapper XML 文件

在 Spring Boot 中配置 XML 路径（application.yml）：

mybatis-plus:
  mapper-locations: classpath*:mapper/**/*.xml  # 默认值通常是这个
  type-aliases-package: com.example.demo.entity

📌 默认情况下，MP 会自动加载 classpath*:mapper/**/*.xml，所以如果你把 XML 放在 src/main/resources/mapper/ 下，通常无需额外配置。

✅ 3. XML 高级特性

（1）resultMap（复杂映射）

<resultMap id="UserDeptMap" type="UserVO">
    <id property="id" column="id"/>
    <result property="name" column="name"/>
    <association property="department" javaType="Department">
        <id property="id" column="dept_id"/>
        <result property="name" column="dept_name"/>
    </association>
</resultMap>

<select id="selectWithDept" resultMap="UserDeptMap">
    SELECT u.*, d.id AS dept_id, d.name AS dept_name
    FROM user u
    LEFT JOIN department d ON u.dept_id = d.id
</select>

（2）动态 SQL（<if>, <choose>, <foreach>）

<select id="selectByCondition" resultType="User">
    SELECT * FROM user
    <where>
        <if test="name != null and name != ''">
            AND name LIKE CONCAT(#{name}, '%')
        </if>
        <if test="minAge != null">
            AND age >= #{minAge}
        </if>
    </where>
</select>

（3）复用 SQL 片段（<sql> + <include>）

<sql id="userColumns">
    id, name, age, email
</sql>

<select id="selectAll" resultType="User">
    SELECT <include refid="userColumns"/> FROM user
</select>

✅ 4. 优点与缺点

优点缺点
SQL 可读性强，格式清晰需维护额外 XML 文件
支持复杂映射（resultMap）启动稍慢（需解析 XML）
动态 SQL 表达能力强路径配置错误易导致方法找不到
与 MP 的 Wrapper 完美融合（${ew.sqlSegment}）—

📌 适用场景：企业级开发、复杂查询、多表关联、报表、批量操作等。

三、注解 SQL vs XML SQL 对比总结

维度注解 SQLXML SQL
可读性差（SQL 在字符串中）优秀（格式化、高亮）
动态 SQL需 <script>，难维护原生支持，简洁强大
复杂映射不支持 resultMap完全支持
维护成本低（单文件）中（需管理 XML）
MP 集成可用，但无法用 ${ew.sqlSegment}完美支持 Wrapper 拼接
团队协作SQL 与 Java 混合，不易分离DBA 可直接看 SQL
推荐度⭐⭐（简单场景）⭐⭐⭐⭐⭐（主流）

四、最佳实践建议

  1. 简单单表操作 → 优先用 MP 的 IService / BaseMapper（无需写 SQL）。
  2. 复杂 SQL（JOIN/子查询/聚合） → 必须用 XML。
  3. 临时调试或极简逻辑 → 可用注解 SQL。
  4. 永远不要在注解中写超过 3 行的 SQL。
  5. XML 文件命名规范：{Entity}Mapper.xml，放在 resources/mapper/ 下。
  6. 字段映射：优先用 resultType（VO/Entity 字段匹配列名），复杂嵌套用 resultMap。

五、混合使用示例（MP + XML + Wrapper）

// Mapper 接口
List<UserVO> selectCustom(@Param(Constants.WRAPPER) Wrapper<User> wrapper);
<!-- XML -->
<select id="selectCustom" resultType="UserVO">
    SELECT u.name, d.dept_name
    FROM user u
    LEFT JOIN dept d ON u.dept_id = d.id
    <where>
        ${ew.customSqlSegment}  <!-- 自动注入 Wrapper 条件 -->
    </where>
</select>
// 调用
var wrapper = new LambdaQueryWrapper<User>()
    .eq(User::getStatus, 1)
    .likeRight(User::getName, "张");
userService.getBaseMapper().selectCustom(wrapper);

✅ 这是 MyBatis-Plus 最强大的组合：自定义 SQL + 动态条件构造器。

总结

方式何时用
MP 内置方法80% 单表 CRUD（首选）
XML SQL复杂查询、多表、批量、动态 SQL（主力）
注解 SQL极简查询、临时测试（慎用）

💡 记住： “能用 IService 就不用写 SQL；要写 SQL，优先选 XML。” 这是 MyBatis-Plus 项目的黄金准则。
