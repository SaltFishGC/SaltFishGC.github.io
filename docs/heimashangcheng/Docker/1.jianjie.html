Docker简介

SaltFishGC2026-1-29

简单概念

Docker 是一个开源的容器化平台，用于将应用程序及其依赖（如代码、运行时、库、配置文件等）打包成一个轻量级、可移植的容器（Container），实现“一次构建，到处运行”。

核心特点：

  • 轻量级：容器共享主机操作系统内核，无需像虚拟机那样启动完整操作系统，启动快、资源占用少。
  • 隔离性：每个容器相互隔离，互不影响。
  • 可移植性：镜像可在任何支持 Docker 的环境（Linux、Windows、macOS、云服务器）中运行。
  • 标准化：通过 Dockerfile 定义构建过程，确保开发、测试、生产环境一致。

核心概念：

概念说明
镜像（Image）只读模板，用于创建容器（如 nginx:latest）
容器（Container）镜像的运行实例，可启动、停止、删除（一个镜像可运行多个容器）
Dockerfile文本文件，定义如何构建镜像
仓库（Registry）存储和分发镜像的地方（如 Docker Hub、阿里云 ACR）

简单示例：

# 拉取 Nginx 镜像
docker pull nginx

# 运行容器（后台，端口映射）
docker run -d -p 8080:80 --name my-web nginx

# 访问 http://localhost:8080 即可看到 Nginx 欢迎页

✅ 一句话总结： Docker 让应用像集装箱一样标准化打包、高效运输、随处部署。

基础操作

启动与验证 Docker 服务

在使用 Docker 前，需确保其后台服务（dockerd）正在运行。

  • Linux（systemd 系统）：

    # 启动 Docker 服务
sudo systemctl start docker

# 设置开机自启
sudo systemctl enable docker

# 查看状态
sudo systemctl status docker

  • Windows/macOS：安装 Docker Desktop 后，启动应用程序即可自动运行服务。

验证安装是否成功：

docker --version        # 查看版本
docker info             # 查看详细信息
docker run hello-world  # 运行测试容器

镜像（Image）操作

镜像是容器的模板，只读且可共享。

镜像名称：Repository:Version 镜像名称 + 版本号

拉取镜像

docker pull nginx:latest    # 从 Docker Hub 拉取官方 Nginx 镜像
docker pull ubuntu:22.04    # 指定标签

列出本地镜像

docker images               # 显示所有本地镜像
docker image ls             # 等价命令

输出包含：REPOSITORY、TAG、IMAGE ID、CREATED、SIZE。

删除镜像

docker rmi nginx            # 删除镜像（通过名称或 ID）
docker rmi <IMAGE_ID>       # 强制删除（即使被容器引用）加 -f

构建自定义镜像（Dockerfile）

# 在包含 Dockerfile 的目录下执行
docker build -t myapp:v1 .
  • -t：指定镜像名和标签
  • .：构建上下文路径

载入tar文件

docker load < archive.tar
# 或
docker load -i archive.tar
  • -i：指定输入的归档文件路径（推荐显式使用，更清晰）

保存本地镜像为tar

docker save -o nginx.tar nginx:latest

容器（Container）操作

容器是镜像的运行实例，有生命周期（创建 → 运行 → 停止 → 删除）。

运行容器

# 后台运行（-d），端口映射（-p），命名（--name）
docker run -d -p 8080:80 --name my-nginx nginx

# 交互式运行（-it），进入容器 shell
docker run -it ubuntu /bin/bash

# 自动删除容器（--rm），常用于临时任务
docker run --rm alpine ping -c 3 baidu.com

查看容器

docker ps          # 查看正在运行的容器
docker ps -a       # 查看所有容器（包括已停止的）
docker container ls -a  # 等价命令

停止与启动

docker stop my-nginx    # 优雅停止（发送 SIGTERM）
docker kill my-nginx    # 强制终止（发送 SIGKILL）

docker start my-nginx   # 启动已停止的容器
docker restart my-nginx # 重启容器

关机后docker会有序stop所有正在run的容器，如想要容器在开机后自动重启，则使用命令：

docker update --restart=unless-stopped mysql

进入运行中的容器

# 执行命令（推荐）
docker exec -it my-nginx /bin/bash

# 附加到容器主进程（不推荐，退出可能终止容器）
docker attach my-nginx

查看日志

docker logs my-nginx        # 查看全部日志
docker logs -f my-nginx     # 实时跟踪日志（类似 tail -f）
docker logs --tail 50 my-nginx  # 查看最后 50 行

删除容器

docker rm my-nginx          # 删除已停止的容器
docker rm -f my-nginx       # 强制删除运行中的容器
docker container prune      # 删除所有已停止的容器

运行容器命令详解

我们以 启动一个带自定义配置的 Nginx 容器 为例，逐步拆解 docker run 的核心参数：

docker run -d \
  --name my-nginx \
  --restart unless-stopped \
  -p 8080:80 \
  -v /host/nginx.conf:/etc/nginx/nginx.conf:ro \
  -v nginx-logs:/var/log/nginx \
  --network my-web-net \
  --memory=256m \
  --cpus=0.5 \
  nginx:1.25

下面逐行详细解释每个参数的作用、原理和最佳实践。

-d：后台运行（Detached Mode）

  • 作用：容器在后台运行，不占用当前终端。

  • 对比：

    • 不加 -d：容器前台运行，关闭终端即停止容器（适合调试）。
    • 加 - d：容器作为守护进程运行（适合服务部署）。

  • 等价写法：--detach

  • ✅ 生产环境必加

--name my-nginx：指定容器名称

  • 作用：为容器分配一个易读、唯一的名称（默认是随机字符串如 happy_pasteur）。

  • 优势：

    • 后续操作更直观：docker stop my-nginx 而非 docker stop a1b2c3d4
    • 容器间可通过名称通信（在自定义网络中）

  • ⚠️ 注意：名称必须全局唯一，重复会报错。

--restart unless-stopped：重启策略

  • 作用：定义容器退出后的自动重启行为。

  • 常用选项：

    策略行为
    no（默认）不自动重启
    always总是重启（即使手动 docker stop）
    on-failure[:max-retries]非0退出码时重启
    unless-stopped推荐！ 除非被手动停止，否则 always 重启

  • ✅ 生产服务强烈建议设置，避免服务器重启后服务不可用。

-p 8080:80：端口映射（Publish Ports）

  • 作用：将主机端口映射到容器端口，格式：-p <主机端口>:<容器端口>

  • 本例含义：

    • 主机访问 http://localhost:8080 → 流量转发到容器的 80 端口（Nginx 默认端口）

  • 高级用法：

    -p 80:80                # 直接绑定 80 端口（需 root）
-p 127.0.0.1:8080:80    # 仅本地回环可访问（安全）
-p 8080:80/udp          # 指定协议（TCP/UDP）

  • ⚠️ 注意：多个容器不能绑定同一主机端口。

-v /host/nginx.conf:/etc/nginx/nginx.conf:ro：绑定挂载配置文件

  • 作用：将主机的 Nginx 配置文件挂载到容器内，并设为只读。

  • 参数分解：

    • /host/nginx.conf：主机上的配置文件路径（必须存在！）
    • /etc/nginx/nginx.conf：容器内的目标路径
    • :ro：read-only，防止容器意外修改配置

  • ✅ 最佳实践：

    • 配置文件用 Bind Mount（便于外部管理）
    • 加 :ro 提升安全性

  • ⚠️ 致命陷阱： 若主机文件不存在，Docker 会创建一个空目录而非文件，导致 Nginx 启动失败！

-v nginx-logs:/var/log/nginx：使用命名卷持久化日志

  • 作用：将容器日志目录挂载到 Docker 管理的命名卷 nginx-logs。

  • 优势：

    • 日志数据与容器生命周期解耦（删容器不丢日志）
    • 卷由 Docker 管理，路径统一（/var/lib/docker/volumes/nginx-logs/_data）
    • 可被其他容器共享或备份

  • ✅ 对比 Bind Mount： Volume 更适合生产环境持久化数据，Bind Mount 更适合开发时挂载代码。

--network my-web-net：指定自定义网络

  • 作用：将容器加入名为 my-web-net 的 Docker 网络。

  • 前提

    ：需提前创建网络：

    docker network create my-web-net

  • 优势：

    • 容器间可通过容器名 DNS 解析通信（如 ping my-nginx）
    • 隔离性更好（默认 bridge 网络所有容器互通）

  • ✅ 多容器应用（如 Web + DB）必须使用自定义网络

--memory=256m 和 --cpus=0.5：资源限制

  • 作用：限制容器使用的系统资源，防止单个容器耗尽主机资源。

  • 参数说明：

    • --memory=256m：最多使用 256MB 内存（超限会被 OOM Kill）
    • --cpus=0.5：最多使用 0.5 个 CPU 核心（即 50% 单核性能）

  • 单位支持：

    • 内存：b, k, m, g（如 512m）
    • CPU：小数（0.5）或毫核（500m 表示 0.5 核）

  • ✅ 生产环境必备，实现多租户资源隔离。

nginx:1.25：镜像名称与标签

  • 作用：指定要运行的镜像。

  • 格式：

    [仓库/][镜像名][:标签]
    • nginx：Docker Hub 官方镜像
    • :1.25：明确指定版本（禁止用 latest！）

  • ✅ 最佳实践：

    • 永远指定明确标签（如 1.25 而非 latest）

  • 生产环境使用私有仓库镜像（如 harbor.example.com/nginx:1.25）

补充：其他高频参数

参数说明示例
-e KEY=VALUE设置环境变量-e TZ=Asia/Shanghai(根据镜像的说明文档指定kv配置)
--env-file .env从文件加载环境变量--env-file ./prod.env
--user 1000:1000指定容器内运行用户避免 root 权限风险
--rm容器停止后自动删除适合临时任务
--entrypoint覆盖镜像默认启动命令--entrypoint /bin/sh

数据管理：卷（Volume）与绑定挂载（Bind Mount）

容器默认是临时的，数据随容器删除而丢失。Docker 提供两种持久化方式：

使用 Volume（推荐）

# 创建命名卷
docker volume create my-vol

# 挂载卷到容器
docker run -d -v my-vol:/app/data nginx

# 查看卷
docker volume ls
docker volume inspect my-vol

# 删除卷
docker volume rm my-vol
docker volume prune  # 清理未使用的卷

使用绑定挂载（Bind Mount）

将主机目录直接挂载到容器：

docker run -d -v /host/path:/container/path nginx
# 示例：挂载当前目录到容器
docker run -v $(pwd):/app alpine ls /app

💡 区别：

  • Volume：由 Docker 管理，跨平台兼容性好，适合生产
  • Bind Mount：直接暴露主机路径，适合开发调试

网络（Network）基础

Docker 默认提供三种网络驱动：bridge（默认）、host、none。

查看网络

docker network ls

创建自定义网络

docker network create my-net

容器连接到自定义网络

docker run -d --network my-net --name app1 nginx
docker run -it --network my-net alpine ping app1  # 可通过容器名通信

✅ 优势：自定义网络支持自动 DNS 解析（容器名即 hostname）

端口映射（-p）

-p 8080:80        # 主机8080 → 容器80
-p 127.0.0.1:8080:80  # 仅绑定本地回环

清理与维护

避免磁盘被无用资源占满：

# 删除所有停止的容器
docker container prune

# 删除所有未被使用的镜像
docker image prune -a

# 删除所有未使用的卷
docker volume prune

# 一键清理（谨慎！）
docker system prune -a

⚠️ docker system prune -a 会删除：

  • 所有停止的容器
  • 所有未被容器引用的镜像
  • 所有未使用的网络
  • 所有未使用的构建缓存

常用组合命令示例

  1. 快速启动并进入 Ubuntu

    docker run -it --rm ubuntu:22.04 bash

  2. 运行 MySQL 并持久化数据

    docker run -d \
  --name mysql-db \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -v mysql-data:/var/lib/mysql \
  -p 3306:3306 \
  mysql:8.0

  3. 查看容器 IP 地址

    docker inspect my-nginx | grep IPAddress
# 或
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' my-nginx

挂载

挂载（Mounting）概述

在 Docker 中，挂载（Mount） 是指将主机（Host）上的文件或目录与容器（Container）内的路径进行关联，使得容器可以读写这些数据。 Docker 提供三种挂载方式：

  • Bind Mounts（绑定挂载）：直接映射主机任意路径。
  • Volumes（卷）：由 Docker 管理的存储，位于 Docker 专属目录（如 /var/lib/docker/volumes/）。
  • tmpfs Mounts：仅存于内存，容器停止后自动清除（仅 Linux 支持）。

✅ 核心目的：实现数据持久化、配置共享、开发热更新。

docker stop → docker start挂载的文件/目录自动同步主机当前最新内容，不必重新docker rm了再docker run。

  • 如果只改了挂载的代码/配置 → 只需 docker restart 容器（或让应用 reload）
  • 如果改了镜像（如升级 Nginx 版本）→ 必须 rm + run 或用滚动更新

绑定挂载（Bind Mounts）

绑定挂载将主机上的任意文件或目录直接挂载到容器中。

基本语法

docker run -v /host/path:/container/path [镜像]
# 或使用 --mount（更明确）
docker run --mount type=bind,source=/host/path,target=/container/path [镜像]

示例：挂载代码目录进行开发

# 将当前项目目录挂载到容器的 /app
docker run -it -v $(pwd):/app node:18 sh
# 在容器内修改 /app 下的文件，主机文件同步更新

特点

优点缺点
路径灵活，可挂载任意位置依赖主机目录结构，移植性差
适合开发调试（实时同步）安全风险高（可能暴露敏感路径）
无需预先创建主机路径不存在时会自动创建为目录（易出错！）

⚠️ 注意： 若主机路径不存在，Docker 会静默创建一个目录而非报错。例如：

docker run -v /nonexist:/data alpine touch /data/file

此时 /nonexist 在主机上会被创建为目录，而非你期望的文件！

数据卷（Volumes）

Volume 是 Docker 推荐的持久化方式，由 Docker 引擎管理，与主机文件系统解耦。

创建与使用

# 方式1：运行时自动创建匿名卷
docker run -v /data nginx  # 卷名随机生成

# 方式2：显式创建命名卷（推荐）
docker volume create my-vol

# 挂载命名卷
docker run -v my-vol:/app/data nginx
# 或
docker run --mount type=volume,source=my-vol,target=/app/data nginx

查看与管理

# 列出所有卷
docker volume ls

# 查看卷详情（含主机实际路径）
docker volume inspect my-vol
# 输出示例：
# "Mountpoint": "/var/lib/docker/volumes/my-vol/_data" （后续到指定为止修改代码配置即可）

# 删除未使用的卷
docker volume rm my-vol
docker volume prune  # 清理所有未被容器引用的卷

特点

优点缺点
跨平台兼容（Windows/Linux/macOS 行为一致）无法直接指定主机路径
安全（Docker 管理权限和位置）需通过 docker volume inspect 找实际路径
支持 Volume Drivers（如云存储插件）不适合需要精确控制主机路径的场景
可备份、迁移、共享—

💡 最佳实践：

  • 生产环境用 Volume
  • 开发环境可用 Bind Mount

tmpfs 挂载（仅 Linux）

将数据挂载到内存中，容器停止后自动清除，适用于敏感临时数据（如密码、会话）。

语法

docker run --tmpfs /tmp/cache nginx
# 或
docker run --mount type=tmpfs,destination=/tmp/cache nginx

配置选项（高级）

docker run --tmpfs /run:rw,noexec,nosuid,size=65536k nginx

常用参数：

  • size：限制大小（默认 50% 内存）
  • mode：设置权限（如 mode=1777）
  • noexec / nosuid：安全限制

⚠️ 限制：

  • 仅支持 Linux
  • 不适用于 Windows/macOS（Docker Desktop 会模拟，但非真正 tmpfs）

挂载选项详解（--mount vs -v）

Docker 支持两种挂载语法：-v（简写） 和 --mount（详细）。

特性-v / --volume--mount
语法简洁性✅ 简洁❌ 冗长
可读性⚠️ 参数顺序易混淆✅ 明确指定 type,source,target
功能完整性❌ 不支持 tmpfs 选项✅ 支持所有挂载类型和选项
推荐场景快速命令行操作脚本、Compose 文件、复杂配置

对比示例

# Bind Mount（等价）
docker run -v /host:/container alpine
docker run --mount type=bind,source=/host,target=/container alpine

# Volume（等价）
docker run -v myvol:/data alpine
docker run --mount type=volume,source=myvol,target=/data alpine

# tmpfs（只能用 --mount）
docker run --mount type=tmpfs,target=/tmp alpine

📌 官方建议：

  • 新用户优先使用 --mount（更清晰）
  • 脚本和 docker-compose.yml 中必须用 --mount 风格

挂载常见场景与最佳实践

场景 1：持久化数据库数据

# 使用 Volume（生产推荐）
docker volume create mysql-data
docker run -d \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=123456 \
  mysql:8.0

场景 2：开发时热更新代码

# 使用 Bind Mount（开发推荐）
docker run -it -v $(pwd)/src:/app/src node:18 npm start

场景 3：挂载配置文件

# 挂载单个文件（注意：主机文件必须存在！）
docker run -v $(pwd)/nginx.conf:/etc/nginx/nginx.conf nginx

⚠️ 重要： 挂载单个文件时，若主机文件不存在，Docker 会创建一个目录而非文件，导致容器启动失败！

场景 4：只读挂载（安全加固）

# 防止容器修改主机文件
docker run -v /host/config:/app/config:ro nginx
# 或
docker run --mount type=bind,source=/host/config,target=/app/config,readonly nginx

场景 5：多容器共享数据

# 创建共享卷
docker volume create shared-data

# 容器 A 写入
docker run -v shared-data:/data writer-app

# 容器 B 读取
docker run -v shared-data:/data reader-app

故障排查与注意事项

  1. 权限问题（Linux 常见） 容器内进程 UID 与主机文件权限不匹配 → 挂载后无读写权限。 解决：

    • 启动容器时指定用户：--user $(id -u):$(id -g)
    • 或在 Dockerfile 中调整用户权限

  2. SELinux 限制（CentOS/RHEL） 需添加 :z 或 :Z 标签：

    docker run -v /host/path:/container/path:z nginx

  3. Windows 路径格式 PowerShell 中需转义或使用绝对路径：

    docker run -v C:\my\path:/data nginx  # 注意反斜杠
# 或
docker run -v ${PWD}:/data nginx

  4. Docker Desktop 文件共享设置（macOS/Windows） 首次使用 Bind Mount 前，需在 Docker Desktop 设置中授权目录访问权限。

dockerhub

网址：Docker Hub Container Image Library | App Containerizationopen in new window（开vpn）

Docker Hub 是由 Docker 官方提供的公共容器镜像仓库（Registry）服务，类似于 GitHub 之于代码，是全球最主流的 Docker 镜像托管平台。开发者可以在这里 拉取（pull）公开镜像、推送（push）自己的镜像，并进行团队协作与版本管理。

一般只是用来看文档（注意版本对应）

简单优化

命令别名(docker ps)

默认的ps检查容器运行情况乱得不行，我们可以使用linux的命令别名来优化命令：

进入 .bashrc 中修改配置（ .bashrc 是 root 下的隐藏文件）：

cd /root

vim .bashrc

在 .bashrc 中加入以下内容：

alias dps='docker ps --format "table {{.ID}}\t {{.Image}}\t{{.Ports}}\t{{.Status}}\t{{.Names}}"'

保存退出后重新加载配置：

source /root/.bashrc
