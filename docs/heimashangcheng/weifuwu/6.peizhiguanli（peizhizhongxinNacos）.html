配置中心

SaltFishGC2026-2-8

Nacos（Dynamic Naming and Configuration Service）是阿里巴巴开源的 服务发现 + 配置管理 中间件，作为 Spring Cloud 生态的重要组件，其配置管理（Configuration Management） 功能可替代 Spring Cloud Config，提供动态、集中、实时生效的外部化配置能力。

基本概念

为什么需要配置中心：

传统方式（如 application.yml）Nacos 配置中心
配置写死在项目中，改配置需重新打包部署配置外置，运行时动态修改
多环境（dev/test/prod）需维护多份文件统一管理所有环境配置
无法灰度发布配置支持 配置版本、回滚、灰度
无变更记录和权限控制提供 操作审计、权限管理、监听通知

nacos的配置管理概念：

概念说明
Namespace最高级隔离单位，用于区分不同环境（如 dev / test / prod）或不同租户
Group配置分组，默认 DEFAULT_GROUP，可用于区分项目或环境
Data ID配置的唯一标识，通常格式：${prefix}-${spring.profiles.active}.${file-extension} 例如：user-service-dev.yaml
配置内容支持 Properties、YAML、JSON、TEXT 等格式

📌 推荐结构：

  • Namespace → 环境（dev / prod）
  • Group → 项目或微服务组（如 USER_GROUP）
  • Data ID → 具体服务配置（如 order-service.yaml）

nacos的配置获取流程：

 [应用启动]
     ↓
1. 初始化 Bootstrap Context（或 ConfigData 加载器）
     ↓
2. 根据配置确定要加载的 Data ID 列表：
   - 主配置：${spring.application.name}-${spring.profiles.active}.${file-extension}
   - 共享配置：shared-configs 中的 dataId 列表
     ↓
3. 向 Nacos Server 发起 HTTP GET 请求（/nacos/v1/cs/configs）
   参数：dataId, group, namespace, tenant
     ↓
4. Nacos Server 查询配置（从内存缓存 → 数据库）
     ↓
5. 返回配置内容（字符串）给客户端
     ↓
6. 客户端将配置解析为 PropertySource，注入 Spring Environment
     ↓
7. Spring 创建 DataSource、MyBatis 等 Bean（使用 Nacos 配置）
     ↓
8. 【运行时】启动长轮询（Long Polling）监听配置变更
     ↓
9. 用户在 Nacos 控制台修改配置 → Server 推送变更通知
     ↓
10. 客户端重新拉取最新配置 → 触发 @RefreshScope Bean 重建

使用注意事项

DataID必须加上文件后缀

后缀没加/错误也会导致nacos找不到所需要的配置文件。

不同 Namespace 之间的配置完全隔离，无法互相访问

dev，prop等namespace在服务的bootstrap/application中只能指定一个！不可在同一服务中使用多个不同namespace的配置文件！

使用bootstrap作为引入nacos配置文件

Spring Boot 本身并不强制要求使用 bootstrap.yml，但 Spring Cloud（包括 Nacos Config）为了保证“配置早于应用上下文初始化”，必须使用 bootstrap 阶段加载配置中心的连接信息。

Spring Boot 应用启动流程大致如下：

1. 创建 Bootstrap Context（引导上下文）
   └── 加载 bootstrap.yml / bootstrap.properties
   └── 初始化 Spring Cloud 组件（如 Nacos Client）
   └── 从 Nacos 拉取远程配置

2. 创建 Application Context（主应用上下文）
   └── 加载 application.yml
   └── 创建 @Bean（如 DataSource、MyBatis 等）
   └── 启动 Web 容器

⚠️ 关键点： DataSource、Redis 等 Bean 的创建依赖 application.yml 中的配置， 而这些配置可能来自 Nacos。 所以必须在 Application Context 初始化之前 就连接 Nacos 并拉取配置！

这就需要一个 比 application.yml 更早加载的配置文件 → bootstrap.yml

如果不用**bootstrap.yml**：

  1. Spring 先读 application.yml
  2. 发现 datasource.url 未定义（因为 Nacos 还没连）
  3. 尝试创建 DataSource Bean → 报错：url is null
  4. 根本没机会去连 Nacos！

但其实新版本我们可以直连nacos，只要按以下配置也可以不去准备bootstrap文件：

# application.yml
spring:
  application:
    name: user-service

  config:
    import: 
      - optional:nacos:user-service.yaml  # 从 Nacos 加载配置

  cloud:
    nacos:
      config:
        server-addr: 127.0.0.1:8848
        group: DEFAULT_GROUP
        namespace: ""  # public 命名空间

老版本不能在一个上下文中完成，是因为 Nacos 客户端依赖 Spring 容器，而容器创建又依赖配置，形成死锁； 新版本通过 Config Data API 让 Nacos 客户端脱离容器，以“配置加载插件”形式在容器初始化过程中直接拉取配置，从而打破死锁。

新版本装载context流程：

  1. 启动 Application Context
  2. 解析 application.yml，遇到 spring.config.import: nacos:xxx
  3. 暂停常规配置加载，转而：
    • 使用 内置的、非 Spring 容器管理的 Nacos 客户端（纯 Java 工具类）
    • 直接调用 Nacos HTTP API 拉取配置（不依赖 @Bean）
  4. 将拉取的配置作为新的 PropertySource 插入 Environment
  5. 继续完成剩余配置解析和 Bean 创建
特性bootstrap.yml方式application.yml + spring.config.import
Spring Boot 版本所有版本（包括 2.4+）仅 Spring Boot 2.4+
是否需要额外依赖需要 spring-cloud-starter-bootstrap不需要（内置支持）
配置加载顺序Bootstrap Context → Application Context单一上下文，按 import 顺序加载
社区成熟度✅ 非常成熟，文档丰富较新，部分企业尚未迁移
推荐场景传统项目、稳定生产环境新项目、拥抱 Spring Boot 新特性

当然在新版本中使用bootstrap也是可以的，更加保险一点

配置与使用

Nacos准备配置文件

访问8848，我们先新建一个jdbc的统一配置文件：

spring:
  datasource:
    url: jdbc:mysql://${hm.db.host:192.168.0.200}:${hm.db.port:3306}/${hm.db.database}?useUnicode=true&characterEncoding=UTF-8&autoReconnect=true&serverTimezone=Asia/Shanghai
    driver-class-name: com.mysql.cj.jdbc.Driver
    username: ${hm.db.un:root}
    password: ${hm.db.pw:123}
mybatis-plus:
  configuration:
    default-enum-type-handler: com.baomidou.mybatisplus.core.handlers.MybatisEnumTypeHandler
  global-config:
    db-config:
      update-strategy: not_null
      id-type: auto

${...} 占位符若带上${path:value}中的:value，其中的value就是默认值（不显示声明自动使用默认值）

这段配置中的 ${...} 占位符（包括默认值如 ${hm.db.host:192.168.0.200}）是由 Spring Boot 的配置解析机制完成的，而不是由 Nacos 或其他外部组件处理的。

再搞一个log的统一配置文件：

logging:
  level:
    com.hmall: debug
  pattern:
    dateformat: HH:mm:ss:SSS
  file:
    path: "logs/${spring.application.name}"

和swagger的统一配置：

knife4j:
  enable: true
  openapi:
    title: ${hm.swagger.title:黑马商城接口文档}
    description: ${hm.swagger.description:黑马商城接口文档}

这样我们就完成了Nacos配置中心的配置文件准备，接下来我们完成SpringBoot项目的配置：

引入依赖

在父项目pom中做好依赖版本管理：

<!--spring cloud-->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-dependencies</artifactId>
    <version>${spring-cloud.version}</version>
    <type>pom</type>
    <scope>import</scope>
</dependency>
<!--spring cloud alibaba-->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-alibaba-dependencies</artifactId>
    <version>${spring-cloud-alibaba.version}</version>
    <type>pom</type>
    <scope>import</scope>
</dependency>

注意alibaba与cloud本体版本映射关系！

然后再在需要的子项目pom中直接引入依赖即可：

  <!--nacos配置管理-->
  <dependency>
      <groupId>com.alibaba.cloud</groupId>
      <artifactId>spring-cloud-starter-alibaba-nacos-config</artifactId>
  </dependency>
  <!--读取bootstrap文件-->
  <dependency>
      <groupId>org.springframework.cloud</groupId>
      <artifactId>spring-cloud-starter-bootstrap</artifactId>
  </dependency>

本地配置文件编写

首先是引入nacos配置的bootstrap文件：

spring:
  application:
    name: cart-service # 服务名称
  profiles:
    active: dev
  cloud:
    nacos:
      server-addr: 192.168.0.200:8848 # nacos地址
      config:
        file-extension: yaml # 文件后缀名
        shared-configs: # 共享配置
          - dataId: shared-jdbc.yml # 共享mybatis配置
          - dataId: shared-log.yml # 共享日志配置
          - dataId: shared-swagger.yml # 共享日志配置

不声明命名空间namespace以及组group则使用默认值：

配置项默认值说明
Namespacepublic（实际 ID 为空字符串 ""）所有未指定 namespace 的客户端都连到 public
GroupDEFAULT_GROUPNacos 配置的默认分组

等价于：

spring:
  cloud:
    nacos:
      server-addr: 192.168.150.101
      config:
        namespace: ""                # ← 默认是 public（ID 为空）
        group: DEFAULT_GROUP         # ← 默认分组
        file-extension: yaml
        shared-configs:
          - dataId: shared-jdbc.yml
            group: DEFAULT_GROUP     # ← 每个 shared-config 也默认用 DEFAULT_GROUP
          - dataId: shared-log.yml
            group: DEFAULT_GROUP
          - dataId: shared-swagger.yml
            group: DEFAULT_GROUP

然后是基本的application文件：

server:
  port: 8082
feign:
  okhttp:
    enabled: true # 开启OKHttp连接池支持
hm:
  swagger:
    title: 购物车服务接口文档
    package: com.hmall.cart.controller
  db:
    database: hm-cart

然后就可以让SpringBoot项目从Nacos中获取配置了。

配置热更新

业务中我们常常会使用一些写在yml等配置文件里的参数用于决定阈值等数据，随后使用@Value，@ConfigurationProperties以及Environment等方式直接注入到指定Bean对象中存储以及使用。但是这样的操作在本地使用时一旦应用运行起来我们就只能一个参数用到重启了，有没有不需要重启也可以实现关键参数变更的操作？

我们可以结合nacos提供的配置管理+@ConfigurationProperties + @RefreshScope来实现统一的配置管理：

首先在nacos完成参数的指定：

# nacos中的cart-service.yml文件
hm:
 cart:
  maxItems: 10

然后再在对应SpringBoot项目中加入新配置文件的声明：

spring:
  application:
    name: cart-service # 服务名称
  profiles:
    active: dev
  cloud:
    nacos:
      server-addr: 192.168.0.200:8848 # nacos地址
      config:
        file-extension: yaml # 文件后缀名
        shared-configs: # 共享配置
          - dataId: shared-jdbc.yml # 共享mybatis配置
          - dataId: shared-log.yml # 共享日志配置
          - dataId: shared-swagger.yml # 共享日志配置
          - dataId: cart-service.yml # 服务配置

我们就可以从applicationContext上下文中获取到数据了：

@Component
@ConfigurationProperties(prefix = "hm.cart")
@RefreshScope
@Data
public class CartProperties {
    // 最大购物车数量
    private Integer maxItems;
}

properties类在业务中注入即可使用数据了：

@Service
@RequiredArgsConstructor
public class CartServiceImpl extends ServiceImpl<CartMapper, Cart> implements ICartService {
    private final CartProperties cartProperties;
    
    ......
        
    private void checkCartsFull(Long userId) {
        int count = lambdaQuery().eq(Cart::getUserId, userId).count();
        if (count >= cartProperties.getMaxItems()) { // 直接获取到数据
            throw new BizIllegalException(StrUtil.format("用户购物车课程不能超过{}", 10));
        }
    }
}

基于nacos的数据配置文件在使用nacos更新时会直接推送到使用该配置文件的服务中，该服务不用重启即可更新对应数据。
