微服务网关

SaltFishGC2026-2-3

一个请求的完整旅程： Client → Nginx（反向代理） → Gateway（路由+过滤） → Nacos（服务发现） → LoadBalancer（选实例） → 目标服务 → （可能嵌套调用） → 响应原路返回

[Client]
   │
   ▼
[Nginx 反向代理]
   │
   ▼
[Spring Cloud Gateway] ←→ [Nacos (注册中心 + 配置中心)]
   │
   ▼
[LoadBalancer 选择实例]
   │
   ▼
[目标微服务（如 order-service）]
   │
   ▼
[可能调用其他微服务（通过 OpenFeign）]
   │
   ▼
[返回响应 → Gateway → Client]

涉及的组件：

组件角色
Nacos服务注册发现 + 配置中心（动态更新路由）
OpenFeign服务间声明式调用
Spring Cloud GatewayAPI 网关，路由 + 过滤 + 协议转换
Spring Cloud LoadBalancer替代 Ribbon，实现客户端负载均衡
Seata分布式事务
Resilience4j / Sentinel熔断、降级、限流
Sleuth + Zipkin分布式链路追踪

Springcloud Gateway

说明文档：Spring Cloud Gatewayopen in new window

Spring Cloud Gateway 是 Spring Cloud 官方推出的新一代 API 网关，用于替代 Netflix Zuul（已停止维护）。它基于 Project Reactor + WebFlux + Netty 构建，是一个高性能、响应式、非阻塞的网关框架。

🎯 核心目标：为微服务架构提供统一入口，实现 路由转发、安全控制、流量治理、可观测性 等能力。

在微服务架构中，存在以下痛点：

  • 每个服务有独立地址，客户端需记住多个 URL
  • 跨域、认证、限流、日志等逻辑重复编写
  • 服务动态扩缩容，IP 不固定
  • 外部直接访问内部服务，存在安全风险

✅ 网关的作用：

  • 统一入口：所有请求先经过网关
  • 隐藏内部细节：客户端只与网关交互
  • 横切关注点集中处理：认证、限流、日志、监控等

关键内容

1.Route（路由）

  • 网关的基本构建块

  • 包含：ID、目标 URI、断言集合、过滤器集合

  • 示例：

    routes:
  - id: user-service # 唯一规则id，自定义即可
    uri: lb://user-service # 目标微服务实例名称，lb代表负载均衡
    predicates: # 断言，会截取以下路径发给规则指定的微服务实例
      - Path=/api/user/**
    filters: # 过滤器，可对传入路径作重写等操作，处理完成后发送给指定实例
      - StripPrefix=1

id随意即可。

这里uri的 user-service 就是 Nacos（或其他注册中心）中注册的服务名称，也就是目标微服务在 application.yml 中配置的：

spring:
application:
 name: user-service   # ← 这个名字！

  • lb 是 LoadBalancer 的缩写

  • 它告诉 Spring Cloud Gateway：

    “不要直接连接一个固定 IP，而是去服务注册中心（如 Nacos）查找名为 user-service 的所有实例，然后通过负载均衡选择一个来转发请求。”

整个流程如下：

  1. Gateway 收到请求：GET /api/user/123

  2. 匹配到路由 Path=/api/user/**

  3. 看到 uri: lb://user-service

  4. 向 Nacos 查询：有哪些实例注册了 spring.application.name = user-service

    • Nacos 返回：[192.168.1.10:8080, 192.168.1.11:8080]

  5. Gateway 使用 Spring Cloud LoadBalancer（默认轮询）选择一个实例，比如 192.168.1.10:8080

  6. 转发请求到：http://192.168.1.10:8080/user/123（注意路径已被 StripPrefix=1 去掉 /api）

2.Predicate（断言）

  • 决定请求是否匹配某条路由
  • 内置多种断言：
    • Path=/api/**
    • Method=GET
    • Header=X-Token, \d+
    • Query=username, abc
    • Host=*.example.com
    • After=2026-01-01T00:00:00Z

3.Filter（过滤器）

  • 在请求转发前后执行逻辑
  • 分两类：
    • GatewayFilter：作用于单个路由
    • GlobalFilter：作用于所有路由

常见过滤器：

过滤器作用
AddRequestHeader添加请求头
StripPrefix去掉路径前缀
RewritePath重写路径
RequestRateLimiter限流（需 Redis）
TokenRelay透传 OAuth2 Token

过滤器filter

大致类型：

GatewayFilter（局部过滤器）

  • 作用范围：仅对单个路由生效

  • 配置方式：在 routes 下的 filters 中声明

  • 示例：

    routes:
  - id: user-service
    uri: lb://user-service
    predicates:
      - Path=/api/user/**
    filters:
      - StripPrefix=1          # ← 局部过滤器
      - AddRequestHeader=X-Source, gateway

GlobalFilter（全局过滤器）

  • 作用范围：对所有路由生效

  • 配置方式：通过 @Component 注册 Bean

  • 示例：

    @Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 全局鉴权逻辑
    }
    @Override
    public int getOrder() { return -100; }
}

🔑 关键区别：

  • GatewayFilter 需显式配置到路由
  • GlobalFilter 自动应用于所有请求

具体类型：

过滤器示例作用
StripPrefix- StripPrefix=1去掉路径前 N 段 /api/user/123 → /user/123

网络登录校验

Gateway部分请求处理流程

Client
   ↓
[HttpServer] (Netty)
   ↓
[Gateway Handler Mapping] → 路由映射，匹配 Route（断言），并通过WebHandler将请求交给指定的路由过滤器组
   ↓
[Filter Chain]
   ├── GlobalFilter 1 (pre)
   ├── GlobalFilter 2 (pre)
   ├── RouteFilter A (pre)
   ├── RouteFilter B (pre)
   ↓
[Proxy Request to Target Service]  ← NettyRoutingFilter（核心转发器，将请求交给指定的微服务实例）
   ↓
   ├── RouteFilter A (post)
   ├── RouteFilter A (post)
   ├── GlobalFilter 2 (post)
   └── GlobalFilter 1 (post)
   ↓
[Write Response to Client]

Request打到网关后根据断言分配到指定的过滤器组，完成过滤器中的pre操作后，交给NettyRoutingFilter根据Nacos获取到的微服务实例通过loadBalance负载均衡到几个实例中，响应结果传回网关，完成过滤器的post操作，在传回nginx/前端。

我们如果需要验证用户信息，就需要在Gateway的过滤器组中安排一个过滤器的pre部分完成验证操作。

自定义GlobalFilter

全局过滤，所有路由都会被处理，声明后自动生效，只需实现GlobalFilter即可：

@Component		// @Component声明并实现GlobalFilter即注册完成全局过滤器
public class MyGlobalFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 获取请求头
        ServerHttpRequest request = exchange.getRequest();
        HttpHeaders headers = request.getHeaders();
        System.out.println(headers);
        // 放行
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        // 设置优先级，值越小优先级越高
        return 0;
    }
}

可以看到我们通过传给过滤器的请求上下文ServerWebExchange拿到了请求ServerHttpRequest，然后完成相关操作后就可以return chain.filter(exchange);交给下一位过滤器了。

自定义GatewayFilter

只在路由配置中声明时作用于该路由，需要在路由中声明。

继承AbstractGatewayFilterFactory<Config>后重写apply方法：return一个重写了filter方法的过滤器。

最后在yml中配置好对应的路由route即可。

@Component
public class PrintAnyGatewayFilterFactory // 父类泛型是内部类的Config类型，注意后缀必须是GatewayFilterFactory
                extends AbstractGatewayFilterFactory<PrintAnyGatewayFilterFactory.Config> {

    @Override
    public GatewayFilter apply(Config config) {
        // OrderedGatewayFilter是GatewayFilter的子类，包含两个参数：
        // - GatewayFilter：过滤器
        // - int order值：值越小，过滤器执行优先级越高
        return new OrderedGatewayFilter(new GatewayFilter() {
            @Override
            public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
                // 获取config值
                String a = config.getA();
                String b = config.getB();
                String c = config.getC();
                // 编写过滤器逻辑
                System.out.println("a = " + a);
                System.out.println("b = " + b);
                System.out.println("c = " + c);
                // 放行
                return chain.filter(exchange);
            }
        }, 100); // 优先级
    }

    // 自定义配置属性，成员变量名称很重要，下面会用到
    @Data
    static class Config{
        private String a;
        private String b;
        private String c;
    }
    // 将变量名称依次返回，顺序很重要，将来读取参数时需要按顺序获取
    @Override
    public List<String> shortcutFieldOrder() {
        return List.of("a", "b", "c");
    }
    // 返回当前配置类的类型，也就是内部的Config
    @Override
    public Class<Config> getConfigClass() {
        return Config.class;
    }

}

注意：该类的名称一定要以GatewayFilterFactory为后缀！

然后就可以注册到yml中了：

spring:
  cloud:
    gateway:
      default-filters:
            - PrintAny=1,2,3 # 注意，这里多个参数以","隔开，将来会按照shortcutFieldOrder()方法返回的参数顺序依次复制

当然指明了参数字段名之后，参数可以直接一对一赋值：

spring:
  cloud:
    gateway:
      default-filters:
            - name: PrintAny
              args: # 手动指定参数名，无需按照参数顺序
                a: 1
                b: 2
                c: 3

实现登录校验

一般直接使用GlobalFilter即可，后面再exclude掉一些不需要的uri路径就行：

@Component
@RequiredArgsConstructor
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    // 记录不需要验证的uri路径
    private final AuthProperties authProperties;

    // jwt秘钥生成以及解析器
    private final JwtTool jwtTool;

    private final AntPathMatcher antPathMatcher = new AntPathMatcher();

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        // 判断是否需要过滤
        if (isExcludePath(request.getURI().getPath())) {
            // 不需要过滤
            return chain.filter(exchange);
        }
        // 取出请求头
        HttpHeaders headers = request.getHeaders();
        // 获取
        String token = headers.getFirst("Authorization");
        if (token == null) {
            // 不存在
            ServerHttpResponse response = exchange.getResponse();
            response.setStatusCode(HttpStatus.UNAUTHORIZED);
            return response.setComplete();
        }

        // 解析
        Long userId = null;
        try {
            userId = jwtTool.parseToken(token);
        }catch (UnauthorizedException e){
            // 解析失败，拦截
            ServerHttpResponse response = exchange.getResponse();
            response.setStatusCode(HttpStatus.UNAUTHORIZED);
            return response.setComplete();
        }

        return chain.filter(exchange);
    }

    /**
     * 判断是否在忽略列表中
     * @param path 请求路径
     * @return 是否在忽略列表中
     */
    private boolean isExcludePath(String path) {
        for (String excludePath : authProperties.getExcludePaths()){
            if (antPathMatcher.match(excludePath, path)){
                return true;
            }
        }
        return false;
    }

    @Override
    public int getOrder() {
        return 0;
    }
}

完成了用户信息的验证后，我们需要将解析出来的信息传给后续的微服务，可以考虑将数据存储到Request的header中，在请求的上下文ServerWebExchange中调用mutate()方法生成一个新的上下文，其的请求头包含用户数据，将这个上下文传给过滤器链，使得下游过滤器以及微服务收到的请求header含所传数据。

mutate() 是 Reactor 框架中 ServerWebExchange 提供的一个方法，用于构建一个新的 ServerWebExchange 实例，同时保留原始对象的状态。

// 添加用户信息
String userInfo = userId.toString();
ServerWebExchange user = exchange.mutate().request(builder -> {
    builder.header("userInfo", userInfo);
}).build();

return chain.filter(user);

现在后续的请求头就包含了用户数据了，接下来我们完成将用户数据存至Threadlocal方便在业务中获取，一般谈到存储请求头数据到tl中就是用mvc的拦截器了（aop也可以），由于基本所有微服务都需要将请求头中的数据存到tl中，我们需要在一个公共模块中完成对拦截器的编写并注入到配置中：

public class UserInfoInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        String userInfo = request.getHeader("userInfo");
        // 获取用户信息，放入线程变量
        if (StrUtil.isNotBlank(userInfo)){
            UserContext.setUser(Long.valueOf(userInfo));
        }
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        UserContext.removeUser();
    }
}

完成拦截器的编写后，我们将之注册到mvcconfig中：

@Configuration
@ConditionalOnClass(DispatcherServlet.class)
public class MvcConfig implements WebMvcConfigurer {

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new UserInfoInterceptor());
    }
}

这里需要注意：

  1. 我们是在common模块中编写的@Configuration文件，默认情况下，其他模块是不会扫描这个文件夹下的@Component文件的！

那么我们就需要做一些额外配置：我们可以在common模块的resources文件夹下的META-INF文件夹内编写一个配置文件spring.factories以实现SpringBoot的自动装配：

org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  com.hmall.common.config.MyBatisConfig,\
  com.hmall.common.config.JsonConfig,\
  com.hmall.common.config.MvcConfig

spring.factories文件是 Spring Boot 中用于自动配置的核心机制之一。它的主要作用是告诉 Spring Boot 在启动时需要加载哪些自动配置类。

  • 通常位于 META-INF/spring.factories 路径下。
  • Spring Boot 启动时会扫描所有依赖中的 spring.factories 文件，读取其中定义的配置类。

org.springframework.boot.autoconfigure.EnableAutoConfiguration 是 Spring Boot 自动配置的入口键。 后面列出的类（如 MyBatisConfig、JsonConfig、MvcConfig）是自定义的配置类，Spring Boot 会在启动时自动加载这些类。 通过这种方式，开发者可以将自己的配置类注册到 Spring Boot 的自动配置流程中。 无需手动在主类上使用 @Import 或 @ComponentScan 注解，Spring Boot 会自动发现并应用这些配置。

这样，我们的config文件就成功配置到了所有微服务模块中。

  1. Gateway模块也引入了common模块，但他是基于webflux的，没有webmvc导致引入mvcConfig会报错！

那么这时候我们就不希望Gateway模块应用mvcConfig配置了，那么这时候我们就需要针对Gateway模块对MvcConfig做一些额外配置：

@ConditionalOnClass(DispatcherServlet.class)

@ConditionalOnClass(DispatcherServlet.class) 是 Spring Boot 提供的一个条件注解，用于控制配置类或 Bean 的加载条件。具体作用如下：

条件判断：只有当 DispatcherServlet 类存在于当前项目的 classpath 中时，被该注解标记的配置类（如 MvcConfig）才会生效。 避免错误：如果项目中没有引入 Web 相关依赖（例如 spring-webmvc），则 DispatcherServlet 类不会存在，此时 MvcConfig 不会被加载，从而避免因缺少依赖导致的启动失败。

但是我们说Gateway也引入了common模块，common模块要配置mvc那肯定是带上了mvc的依赖的，为什么说Gateway没有mvc的依赖呢？因为common模块中的依赖一般都是仅编译时引入（写代码不报错），运行不引入的（其他模块需要自行引入相关依赖）：

<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-webmvc</artifactId>
    <scope>provided</scope>
</dependency>

<scope>provided</scope>：provided作用域表示仅在编译时引入

而Gateway没有显式引入mvc依赖，那么其实运行起来的时候是没有mvc依赖的，也就不会配置mvcConfig了。

Openfeign请求带上用户信息

我们先前完成了将前端发来的jwt在Gateway进行解析并存入请求头，然后再在微服务中将请求头中的解析完成的数据装配到Threadlocal中。但是这只是Gateway到微服务的http请求头会带上用户数据，微服务之间还会进行Openfeign相互调用，这时候如果不做额外配置，Openfeign发送的http请求头是不会带上需要的用户信息的。

Openfeign其实就提供了一个拦截器接口RequestInterceptor，帮助我们完成请求头等信息的处理：

首先在公共Openfeign-api模块中加入新配置：

public class DefaultFeignClient {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return new RequestInterceptor() {
            @Override
            public void apply(RequestTemplate requestTemplate) {
                Long userId = UserContext.getUser();
                if (userId != null) {
                    requestTemplate.header("userInfo", userId.toString());
                }
            }
        };
    }
}

然后在其他模块中使用这个配置：

@MapperScan(basePackages = "com.hmall.item.mapper")
@SpringBootApplication
@EnableFeignClients(basePackages = "com.hmall.api.client", defaultConfiguration = DefaultFeignClient.class)
public class ItemServiceApplication {

    public static void main(String[] args) {
        SpringApplication.run(ItemServiceApplication.class, args);
    }

}

那么这个微服务实例的业务处理线程就会在Openfeign发送RPC请求时将Threadlocal中的用户数据写入请求头，然后其他微服务实例的线程在处理这样的http请求时会使用拦截器对这个请求进行截取，将请求头中的数据拿出并写入自己的业务处理线程的Threadlocal中。

EX：JWT验证

用户登录 → 后端生成一个加密字符串（JWT）→ 前端保存 → 每次请求带上它 → 后端解密验证 → 知道是谁在访问

全程不存 Session（不存Redis），无状态！

类名职责（一句话说清）
JwtUtilJWT 工具箱： • 生成 Token • 解析用户名 • 验证是否过期/被篡改
CustomUserDetailsService用户加载器： 从数据库查用户（用于登录校验 & Token 验证时比对）
AuthController登录入口： 接收账号密码 → 调 Security 校验 → 成功就调 JwtUtil 生成 Token 返回
JwtRequestFilter守门员（关键！）： 每个请求进来先看有没有 Token → 有就解析 → 验证合法 → 把用户身份塞进 Spring Security 上下文
SecurityConfig安全总开关： • 关掉 Session（设为 STATELESS） • 放行 /login • 注册 JwtRequestFilter 到过滤器链

单体项目下我们就只需要在拦截器中完成jwt的解析即可。

微服务中我们需要在网关完成相应的操作：

EX：Spring从配置文件中获取数据

前提：配置文件格式（YAML）

# application.yml
app:
  name: MySpringBootApp
  version: 1.0.0
  debug-mode: true
  timeout-ms: 5000

database:
  url: jdbc:mysql://localhost:3306/mydb
  username: root
  password: secret
  pool:
    max-active: 20
    min-idle: 5

方法一：使用 @Value 注解（简单属性）

适用于读取单个配置项。

import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

@Component
public class AppConfig {

    @Value("${app.name}")
    private String appName;

    @Value("${app.timeout-ms}")
    private int timeoutMs;

    @Value("${database.url}")
    private String dbUrl;

    // 注意：YAML 中的连字符（-）在 @Value 中需用原样写
    @Value("${app.debug-mode}")
    private boolean debugMode;

    // getter...
}

⚠️ 注意：

  • 如果配置不存在，启动会报错（除非设置默认值）
  • 默认值写法：@Value("${app.name:DefaultName}")

方法二：使用 @ConfigurationProperties（推荐！对象绑定）

适用于结构化配置（如数据库、第三方服务等），类型安全、支持校验、可复用。

步骤 1：创建配置类

@Component
@ConfigurationProperties(prefix = "database") // 对应 yml 中的 database:
public class DatabaseProperties {

    @NotEmpty
    private String url;
    
    @NotEmpty
    private String username;
    
    private String password;

    private Pool pool = new Pool();

    // nested class for pool config
    public static class Pool {
        @Min(1)
        private int maxActive = 10;
        
        @Min(0)
        private int minIdle = 1;

        // getters & setters
        public int getMaxActive() { return maxActive; }
        public void setMaxActive(int maxActive) { this.maxActive = maxActive; }
        public int getMinIdle() { return minIdle; }
        public void setMinIdle(int minIdle) { this.minIdle = minIdle; }
    }

    // getters & setters for top-level fields
    public String getUrl() { return url; }
    public void setUrl(String url) { this.url = url; }
    public String getUsername() { return username; }
    public void setUsername(String username) { this.username = username; }
    public String getPassword() { return password; }
    public void setPassword(String password) { this.password = password; }
    public Pool getPool() { return pool; }
    public void setPool(Pool pool) { this.pool = pool; }
}

或使用lombok的@Data

步骤 2：启用 ConfigurationProperties（Spring Boot 2.2+ 可省略）

从 Spring Boot 2.2 开始，只要类上有 @Component，就自动生效。 如果你用的是老版本，或想集中管理，可在主类加：

@EnableConfigurationProperties(DatabaseProperties.class)

步骤 3：使用

@Service
public class UserService {
    
    // 构造器注入，可使用
    private final DatabaseProperties dbProps;

    public UserService(DatabaseProperties dbProps) {
        this.dbProps = dbProps;
    }

    public void connect() {
        System.out.println("Connecting to: " + dbProps.getUrl());
        System.out.println("Max active: " + dbProps.getPool().getMaxActive());
    }
}

✅ 优点：

  • 自动类型转换（String → int, boolean 等）
  • 支持嵌套对象（如 pool.max-active）
  • 支持 JSR-303 校验（需引入 spring-boot-starter-validation）
  • IDE 支持（如 IntelliJ 能自动提示配置）

方法三：使用 @PropertySource（加载自定义 YAML 文件）

默认 Spring Boot 只加载 application.yml。如果你想加载额外的配置文件：

@Configuration
@PropertySource("classpath:custom-config.yml")
// ❌ 注意：@PropertySource 默认不支持 YAML！

⚠️ 问题：@PropertySource 原生只支持 .properties，不支持 YAML！

解决方案：改用 .properties，或通过 Environment 手动加载（不推荐）

✅ 推荐做法：把所有配置放在 application.yml，或使用 Profile 多环境配置（如 application-dev.yml）。

方法四：通过 Environment 接口（动态获取）

适用于工具类或无法注入的场景。

@Component
public class ConfigUtil {

    private final Environment env;

    public ConfigUtil(Environment env) {
        this.env = env;
    }

    public String getAppName() {
        return env.getProperty("app.name", "DefaultApp"); // 第二个参数是默认值
    }

    public Integer getTimeout() {
        return env.getProperty("app.timeout-ms", Integer.class, 3000);
    }
}

✅ 优点：灵活，支持默认值和类型转换 ❌ 缺点：没有编译时检查，容易拼错 key

场景推荐方式
读取 1~2 个简单配置@Value
读取一组结构化配置（如数据库、Redis、第三方 API）@ConfigurationProperties ✅（首选）
动态获取 or 工具类Environment
加载非 application 的配置文件合并到 application.yml 或使用 Profile
