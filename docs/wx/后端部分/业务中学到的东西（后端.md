### 对于业务异常的处理

当出现重复，输入参数不对时，我们需要将提示信息包装到response中再返回给前端，假如是在Service层遇到的，那么我们还需要将出错类型与传参作映射（直接传字符串不美观（为什么感觉有点蠢？））。这样非常的不方便，我们可以使用自行设计业务异常BusinessException的方式来实现对错误的处理，这样还不用在Service层中出错时必须传输错误详细信息，只需要配置好全局异常捕获即可简单实现效果。

**导向：**`@RestControllerAdvice`与`@ExceptionHandler(value = Exception.class)`注解的使用；`BusinessException`的设计

### 常量尽量不要硬编码

硬编码，就是写死，比如数据库连接字符串，用户名，密码，数据库名等等，这样一旦需要修改，就需要去修改代码，这样非常不方便，我们可以使用常量来定义，这样只需要修改常量即可，这样非常方便。

比如我们定义了一个常量类Constants，将一些常量以public static final的静态变量的形式定义在里面，这样我们就可以在任何地方引用这个常量，这样非常方便后续维护修改。

### 能尽量捕获多的异常尽力去使用自己的全局异常捕获

不是自己包装的异常直接抛出不安全，且不美观。

**导向：**`@RestControllerAdvice`+`@ExceptionHandler(value = Exception.class)`全局异常捕获的使用

### 为什么需要关联查询

在写业务的时候我们会发现一些时候多加一点字段能少联几次表，为什么一定要关联查询呢？

因为有些字段的内容可能会修改，比如nickname昵称，签名，等等，假如我们在这里直接存死了，那么下一次修改的时候，就需要把这个表一并修改，倘若一不小心忘了，就成了比较若智的bug了。

所以一般把一行数据里面绝对不会变的一个数据与其他表作关联，其他有可能会修改的数据，我们以多联表查询的形式来导出，防止多处修改使得业务变得不必要的冗余复杂（解耦）。

### 为什么需要长连接

直接看应用场景：

| 场景            | 短连接（轮询）             | 长连接（WebSocket）              |
| --------------- | -------------------------- | -------------------------------- |
| 聊天应用        | 客户端每2秒查一次数据库    | 建立连接后，对方发消息你立刻收到 |
| 股票行情        | 每秒发10次请求查价格       | 服务器主动推送最新价格           |
| 在线游戏        | 按“刷新”按钮才更新         | 角色移动、攻击实时同步           |
| 设备监控（IoT） | 设备定时上报               | 服务器可随时下发指令             |
| 订单状态通知    | 用户刷新页面看是否支付成功 | 支付成功后服务端立刻推“支付完成” |

可以发现仅支持处理客户端发来的request的后端服务器只能支持轮询来实现与各个客户端的通信，这样的方式存在出多弊端，而长连接就较好的解决了这些问题：

| 痛点             | 长连接如何解决              |
| ---------------- | --------------------------- |
| **延迟高**       | 消息直达，无需等待轮询周期  |
| **资源浪费**     | 1次连接 vs 数百次 HTTP 请求 |
| **数据库压力大** | 不用频繁查表                |
| **无法主动通知** | 服务端可以随时“喊一声”      |
| **用户体验差**   | 用户不用刷新就能看到更新    |

当然，这样的长连接也并非完全是好处，所以请结合业务场景来合理选用对应的解决方法：

| 代价         | 说明                               | 如何应对                     |
| ------------ | ---------------------------------- | ---------------------------- |
| 内存占用     | 每个连接占用一定内存（如 4KB~8KB） | 使用连接池、及时关闭无用连接 |
| 连接管理复杂 | 需处理断线、重连、心跳             | 使用 Netty、WebSocket 协议栈 |
| 集群难度高   | 用户连接分布在不同服务器           | 需要 Redis、MQ 做消息广播    |
| 超大规模挑战 | 百万连接需优化内核参数             | 使用 epoll、协程等技术       |

**导向：**长连接；WebSocket；低延迟要求的实现

### 事务开启后注意读写先后顺序

一般来说注意尽量先读后写：

- 写操作会加锁（如行锁、间隙锁）。
- 如果先写，锁持有时间变长，增加与其他事务死锁的概率。
- 先读（必要时加 `FOR UPDATE`）再写，可以更精准地控制锁的范围和时间。

### **`ON DUPLICATE KEY UPDATE`**是什么？

MySQL特有的**尝试插入一条记录**，如果发现这条记录的主键（PRIMARY KEY）或唯一键（UNIQUE KEY）已经存在（即会引发“Duplicate entry”错误），那么就**执行更新操作**，而不是报错。

即insertOrUpdate的常用基本sql语句。

这里之所以额外提一嘴是因为mybatisplus的insertOrUpdate是基于你在PO类中所标识的主键进行的，其余的联合索引等他是不做判断的，所以业务复杂的时候还是需要用上这个自己去写mapper的。

**但是请注意**，在高并发场景下，这一条命令会导致死锁的问题：

[INSERT ... ON DUPLICATE KEY UPDATE产生death lock死锁原理_insert on duplicate key会加读锁吗-CSDN博客](https://blog.csdn.net/pml18710973036/article/details/78452688)

[IGNORE，REPLACE，ON DUPLICATE KEY UPDATE在避免重复插入记录时存在的问题及最佳实践_sql设置ignore on duplicate key update还有效果吗-CSDN博客](https://blog.csdn.net/Saintyyu/article/details/104330881)

可能看到由于会获取X锁，导致执行同一行的其他带X锁会与之竞争，而这时如果恰好还就是`ON DUPLICATE KEY UPDATE`，那么就会一直等待对方的S锁释放了才能上X锁进行update操作，这时就会导致死锁！

虽然可以请求时间过长抛出异常回滚事务，但是在**生产环境**还是**尽量避免使用DUPLICATE**，尤其是对**同一行**会进行高并发DUPLICATE操作时！

还是推荐对相关索引检查后分布操作，那么这里如果还是需要对相关行加锁的话，我们可以使用SELECT FOR UPDATE的方式来实现上X锁（直接上X锁，不是分布S再X）

###  **S 锁（共享锁）** 和 **X 锁（排他锁）**

| 锁类型   | 全称           | 中文名        | 允许的操作                                   | 兼容性                 |
| -------- | -------------- | ------------- | -------------------------------------------- | ---------------------- |
| **S 锁** | Shared Lock    | 共享锁 / 读锁 | 其他事务可以**读**该数据（所有事务都不能写） | ✅ 可与其他 S 锁共存    |
| **X 锁** | Exclusive Lock | 排他锁 / 写锁 | 当前事务可以**读+写**该数据                  | ❌ 不能与任何其他锁共存 |

- **多个 S 锁可以同时存在**（多人同时读没问题）
- **X 锁是“霸道”的**：只要有人持有 S 或 X 锁，X 锁就必须等
- **S 锁遇到 X 锁也会等**（因为 X 锁可能正在修改数据，读旧值会不一致）

| 时间 | 事务 A（持有 S 锁）                                          | 事务 B（请求 X 锁）                                         |
| :--- | ------------------------------------------------------------ | ----------------------------------------------------------- |
| t1   | `SELECT balance FROM accounts WHERE id=1 LOCK IN SHARE MODE;` → 读到 100 |                                                             |
| t2   | （正在处理业务逻辑）                                         | `UPDATE accounts SET balance = 200 WHERE id=1;` → 需要 X 锁 |
| t3   |                                                              | **阻塞！等待 A 释放 S 锁**                                  |

注意是等待其他事务释放了S锁后加上X锁。

### 复合主键

当一张表设置了**两个主键**（即复合主键）时，要求是：**这两个字段的组合值不能重复**，但**单个字段的值可以重复**。

比如一个表记录了用户与其他用户之间的关系：

| user_id | contact_id |
| ------- | ---------- |
| 101     | 102        |
| 101     | 103        |

这时userid可以自己重复，contactid也可以同列重复，但是如果两列都重复了那就不行了。

### 多联表与多次查询的使用场景

| 维度               | 多联表查询（JOIN）      | 多次单独查询           |
| ------------------ | ----------------------- | ---------------------- |
| **网络开销**       | ✅ 一次请求              | ❌ 多次请求             |
| **数据库连接占用** | ✅ 少                    | ❌ 多（尤其并发高时）   |
| **SQL 复杂度**     | ❌ 高（难写、难维护）    | ✅ 低（简单清晰）       |
| **索引利用**       | ⚠️ 可能失效（大表 JOIN） | ✅ 容易命中索引         |
| **缓存友好性**     | ❌ 差（结果不可复用）    | ✅ 好（可缓存单表数据） |
| **扩展性**         | ❌ 差（加表就改 SQL）    | ✅ 好（模块化）         |
| **锁竞争**         | ❌ 高（长时间持有锁）    | ✅ 低（短事务）         |

**多联表（JOIN）：**

1. **数据量小，关系明确**

   - 如：用户 + 角色 + 权限 查询
   - 三张小表 JOIN，速度快

2. **需要排序、分页的汇总数据**

   ```sql
   SELECT u.name, c.last_message, g.member_count
   FROM chat_session_user u
   JOIN chat_session c ON c.session_id = u.session_id
   LEFT JOIN group_info g ON g.group_id = u.contact_id
   ORDER BY c.last_receive_time DESC
   LIMIT 10;
   ```

   - 如果分页，必须一次查完，否则无法保证一致性

3. **强一致性要求高**

   - 所有数据来自同一时刻的数据库快照

4. **网络延迟高（如跨机房）**

   - 减少数据库往返次数至关重要

----

 **多次查询 + 应用层处理**：

1. **大表 JOIN 性能差**
   - 比如 `chat_message` 有上亿条记录
   - `JOIN` 可能导致全表扫描、索引失效
2. **部分数据可缓存**
   - 用户信息、群信息 → 可缓存在 Redis
   - 消息列表 → 从数据库查
   - 应用层拼接，减少数据库压力
3. **微服务架构中，数据分布在不同服务**
   - 用户服务、消息服务、群组服务各自独立
   - 只能先查一个，再调 API 获取其他
4. **异步加载需求**
   - 先显示会话列表（快）
   - 再异步加载未读数、成员数等（慢）

### netty中通话channel的设计

每个用户自己会在服务器生成一个channel存储在内存中，根据userid放到Hashmap里；

每个群组会在服务器生成一个channelgroup（当遍历用户群组发现有时会尝试寻找内存中是否有该channelgroup，无则新建一个），根据groupid放到Hashmap里；

（userid和groupid是唯一存在数据库中的）

用户之间发送消息时会向对方的channel发送消息，用户向群组里发送消息会向群组channelgroup内所有在线的channel发送消息。（消息信息会存储到服务器，离线消息会在用户登录后向服务器请求获取离线后的消息）

### 离线后消息的获取

当我们登录时，channel可以直接获取消息并发送给客户端，客户端缓存到本地sqlite，但是离线了怎么办？

事实上，用户发送了消息之后，消息内容会被后端服务器记录（包括发送对象，接收对象，发送时间等）。用户在登录，channel完成创建时，会向数据库发送一个离线消息的查询，会将用户离线后接收到的消息进行查询并返回给客户端。

### Redis消息队列的使用—Redission

消息队列我们都知道一般用来解耦，将一个主线程的多个步骤分解为一个主线程+多个副线程同时处理或是延后处理的异步操作，可以将业务模块化，将各个业务与主线程隔离开来防止阻塞主线程。

在本项目中，消息队列的使用还用于实现多个服务器部署时的消息同步。因为用户发送request到的服务器与其他用户的服务器可能不同，本项目中netty的channel又是只存储在一个后端服务器的内存当中的，这就要求我们实现多个服务器的消息通信，而这也可以由消息队列来解决。

只需要将Redis以单机或者集群（redis集群如Redis Cluster本身就支持节点通信以实现数据一致性）部署完毕后，确保多台后端服务器连接到该redis服务上，就可结合redisson提供的广播实现服务器之间的消息通信了。

```java
/**
 * 监听频道消息
 */
@PostConstruct // 启动时执行监听
public void listenMessage() {
    // 订阅话题频道消息，接收通过redis传输的其他节点的消息
    RTopic topic = redissonClient.getTopic(MESSAGE_TOPIC);
    topic.addListener(MessageSendDto.class, (channel, msg) -> {
        // 接收到消息。根据信息尝试对自己的内存内的receiveId用户发送消息
        logger.info("接收到消息：{}", msg);
        channelContextUtils.sendMessage(msg);
    });
}

/**
 * 向redission频道发送消息
 * @param message 要发送的消息
 */
public void sendMessage(MessageSendDto message) {
    // 发送消息
    RTopic topic = redissonClient.getTopic(MESSAGE_TOPIC);
    topic.publish(message);
}
```

注意redissonClient的使用需要自己配置config：

```java
/**
 * 创建 RedissonClient 对象，用于操作 Redis
 * @return RedissonClient 对象
 */
@Bean(destroyMethod = "shutdown")
public RedissonClient redissonClient() {
    Config config = new Config();
    config.setCodec(new org.redisson.codec.JsonJacksonCodec()).
            useSingleServer().
            setAddress("redis://" + host + ":" + port).
            setDatabase(Integer.parseInt(database));
    return Redisson.create(config);
}
```

这样，当我们的用户channel处于不同的服务器，不同的服务器会监听redission话题频道的这个sendMessage方法。

当用户发送给其他**单一用户**消息时，监听到命令的**所有服务器**会尝试在自己的内存中查找msg给出的contactId的channel，没有则不响应，有则返回结果。

当用户发送给**群组**消息时，监听到命令的**所有服务器**也会尝试在自己的内存中查找msg给出的contactId的channelGroup，并对group内的所有channel发送消息，结果同上。

这样我们就实现了消息的跨服务器传播。

### static和final

`final` 的意思是“最终的”，一旦被修饰，就不能再改变。

 1. `final` 修饰变量

- **基本类型**：值不能改变。
- **引用类型**：引用不能改变（即不能指向另一个对象），但对象内部的属性可以变。

```java
final int a = 10;
a = 20; // ❌ 编译错误！

final List<String> list = new ArrayList<>();
list.add("hello"); // ✅ 可以，对象内容可以变
list = new ArrayList<>(); // ❌ 不行，引用不能变
```

> ✅ 常用于定义常量，尤其是配合 `static` 使用。

---

 2. `final` 修饰方法

- 表示该方法**不能被子类重写（override）**。

```java
class Parent {
    final void print() {
        System.out.println("This cannot be overridden");
    }
}

class Child extends Parent {
    // void print() { } // ❌ 编译错误！
}
```

> ✅ 常用于防止关键逻辑被篡改，比如安全、工具类方法。

---

 3. `final` 修饰类

- 表示该类**不能被继承**。

```java
final class MathUtils {
    public static int add(int a, int b) {
        return a + b;
    }
}

// class MyMath extends MathUtils { } // ❌ 编译错误！
```

> ✅ 典型例子：`String`、`Integer` 等包装类都是 `final` 的，保证不可变性和安全性。

---

`static` 的意思是“静态的”，它属于类本身，而不是某个实例。

 1. `static` 修饰变量（类变量）

- 所有实例**共享**同一个静态变量。
- 可以通过 `类名.变量名` 直接访问。

```java
class Counter {
    static int count = 0;
    Counter() {
        count++;
    }
}

Counter c1 = new Counter();
Counter c2 = new Counter();
System.out.println(Counter.count); // 输出 2
```

> ⚠️ 注意线程安全问题，多线程环境下需谨慎使用。

---

 2. `static` 修饰方法

- 静态方法属于类，**不能访问非静态成员**（变量或方法）。
- 可以直接通过 `类名.方法名()` 调用。

```java
class MathUtils {
    static int add(int a, int b) {
        return a + b;
    }
}

int result = MathUtils.add(3, 5); // ✅ 直接调用
```

---

 3. `static` 修饰代码块

- 在类加载时执行，只执行一次。
- 常用于初始化静态资源。

```java
static {
    System.out.println("静态代码块执行了");
    // 比如加载配置、初始化缓存等
}
```

---

 4. `static` 内部类

- 普通内部类依赖外部类实例，而 `static` 内部类不依赖。

```java
class Outer {
    static class Inner {
        void print() {
            System.out.println("静态内部类");
        }
    }
}

// 使用：
Outer.Inner inner = new Outer.Inner(); // 不需要 Outer 实例
```

---

 **`final` 和 `static` 一起使用**

最常见的场景是定义**全局常量**：

```java
public class Constants {
    public static final String FILE_FOLDER_FILE = "/file/";
    public static final String FILE_FOLDER_AVATAR_NAME = "avatar/";
}
```

- `static`：属于类，可通过 `Constants.FILE_FOLDER_FILE` 直接访问。
- `final`：值不可变，确保是常量。

#### 小项目：SELECT FOR UPDATE

SELECT FOR UPDATE可以实现对对应行的加锁，在一个事务中执行了这么一条命令后，其他事务想要写或者是带锁的读都会被阻塞：

| 操作类型                                              | 是否被 `SELECT ... FOR UPDATE` 阻塞？ | 说明                                                         |
| ----------------------------------------------------- | ------------------------------------- | ------------------------------------------------------------ |
| **普通 `SELECT`**（如 `SELECT * FROM t WHERE id=1`）  | ❌ **不阻塞**                          | InnoDB 使用 MVCC（多版本并发控制），读的是**历史快照**，无需加锁 |
| **`SELECT ... FOR UPDATE`**                           | ✅ **阻塞**                            | 需要获取 X 锁，会被已持有的 X 锁阻塞                         |
| **`SELECT ... LOCK IN SHARE MODE`**（或 `FOR SHARE`） | ✅ **阻塞**                            | S 锁与 X 锁互斥                                              |
| **`UPDATE` / `DELETE`**                               | ✅ **阻塞**                            | 这些操作会尝试加 X 锁，与已有 X 锁冲突                       |
| **`INSERT`（插入新行）**                              | ❌ **通常不阻塞**                      | 除非涉及唯一键冲突或间隙锁（见下文）                         |

这时我们就可以对相关行进行更新操作了。

sql：

```java
import com.baomidou.mybatisplus.core.mapper.BaseMapper;
import org.apache.ibatis.annotations.Param;
import org.apache.ibatis.annotations.Select;
import java.math.BigDecimal;
import java.util.List;

public interface AccountMapper extends BaseMapper<Account> {

    // 按 user_id 查询（用于 FOR UPDATE）
    @Select("SELECT * FROM account WHERE user_id = #{userId} FOR UPDATE")
    Account selectForUpdateByUserId(@Param("userId") String userId);

    // 按多个 user_id 查询，并按 user_id 排序（避免死锁）
    @Select({
        "<script>",
        "SELECT * FROM account WHERE user_id IN ",
        "<foreach collection='userIds' item='id' open='(' separator=',' close=')'>",
        "#{id}",
        "</foreach>",
        " ORDER BY user_id FOR UPDATE",
        "</script>"
    })
    List<Account> selectForUpdateByUserIds(@Param("userIds") List<String> userIds);
}
```

service：


```java
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.retry.annotation.Backoff;
import org.springframework.retry.annotation.Retryable;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.math.BigDecimal;
import java.util.Arrays;
import java.util.List;

@Slf4j
@Service
@RequiredArgsConstructor
public class TransferService {

    private final AccountMapper accountMapper;

    // 最多重试 3 次，间隔指数退避
    @Retryable(
        value = {Exception.class},
        maxAttempts = 3,
        backoff = @Backoff(delay = 50, multiplier = 2)
    )
    @Transactional(rollbackFor = Exception.class)
    public void transfer(String fromUserId, String toUserId, BigDecimal amount) {
        if (fromUserId.equals(toUserId)) {
            throw new IllegalArgumentException("不能给自己转账");
        }
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException("金额必须大于0");
        }

        // 按固定顺序（字典序）锁定账户，避免死锁
        List<String> userIds = Arrays.asList(
            fromUserId.compareTo(toUserId) < 0 ? fromUserId : toUserId,
            fromUserId.compareTo(toUserId) < 0 ? toUserId : fromUserId
        );

        List<Account> accounts = accountMapper.selectForUpdateByUserIds(userIds);
        if (accounts.size() != 2) {
            throw new RuntimeException("账户不存在");
        }

        Account fromAccount = accounts.get(0).getUserId().equals(fromUserId) 
            ? accounts.get(0) : accounts.get(1);
        Account toAccount = accounts.get(0).getUserId().equals(toUserId) 
            ? accounts.get(0) : accounts.get(1);

        // 检查余额
        if (fromAccount.getBalance().compareTo(amount) < 0) {
            throw new RuntimeException("余额不足");
        }

        // 执行转账
        fromAccount.setBalance(fromAccount.getBalance().subtract(amount));
        toAccount.setBalance(toAccount.getBalance().add(amount));

        accountMapper.updateById(fromAccount);
        accountMapper.updateById(toAccount);

        log.info("转账成功: {} -> {}, 金额: {}", fromUserId, toUserId, amount);
    }
}
```

这里由于存在锁竞争的问题，可能会导致死锁的问题，所以在事务的基础加上了重试以保证尽可能成功。

这里还可以加入乐观锁以保证最大程度的安全。

缺点：高qps下锁竞争激烈，会导致性能下降

### Redis中的hash

不能对单独的一项设置过期时间，只能整体设置过期时间

数据量大时不如简单的list紧凑

**适合用 Hash 存储的数据特征：**

1. **一个逻辑对象包含多个属性字段**
   例如：用户信息（name, age, email）、商品信息（title, price, stock）等。
2. **需要部分更新或读取字段**
   比如只更新用户的邮箱，而不影响其他字段。
3. **字段数量适中（几十到几百个）**
   Redis 对小 Hash 有内存优化（ziplist / listpack 编码），比多个 String 更省内存。
4. **不需要对字段单独设置过期时间**
   Hash 整体共用一个 key 的 TTL，不能为每个 field 单独设过期。

**注意！**Redis中的所有**顶层key**都是使用**hashdict**来实现查找的！所以时间复杂度都是O(1)，不是只有hash使用了hash的！



### Aspect中获取当前线程绑定的request上下文

```java
    private void checkLogin(Boolean checkAdmin) throws BusinessException {
        // 获取当前线程绑定的请求上下文
        ServletRequestAttributes attributes = (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
        HttpServletRequest request = attributes.getRequest();
        String token = request.getHeader("token");
        .....
    }
```

1. 获取请求上下文:
   使用 RequestContextHolder.getRequestAttributes() 获取当前线程绑定的请求上下文
   这是Spring框架提供的机制，可以在任何地方访问当前HTTP请求信息
2. 类型转换:
   将返回的 RequestAttributes 强制转换为 ServletRequestAttributes
   ServletRequestAttributes 是专门用于Servlet环境的请求属性包装器
3. 获取请求对象:
   调用 attributes.getRequest() 方法获得实际的 HttpServletRequest 对象
   通过这个对象可以访问HTTP请求的各种信息，如headers、parameters等

> ServletRequestAttributes: Servlet请求属性的包装器，提供了对Servlet特定功能的访问
> HttpServletRequest: 实际的HTTP请求对象，包含：
> 	请求头信息: 如用于认证的 token 头部
> 	请求参数: 查询参数和表单数据
> 	会话信息: 用户session数据
> 	请求路径: URL路径和HTTP方法
> 	客户端信息: IP地址等
