## AiService

**什么是AiService？**

简单一句话总结：

> **`AiService` 是 LangChain4j 提供的一种“声明式 AI 编程”方式 —— 你只需定义一个 Java 接口，框架自动帮你生成能调用大模型（LLM）的实现类，无需写胶水代码。**

它就像 **MyBatis 的 Mapper 接口** 或 **Spring Data JPA 的 Repository**，但面向的是 **AI 能力**。

### 依赖与配置

说明文档：[AI Services | LangChain4j 中文文档](https://docs.langchain4j.info/tutorials/ai-services)

首先要引入增强的Langchain依赖：

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-spring-boot-starter</artifactId>
</dependency>
```

然后准备一个接口：

```java
public interface Assistant {
    String chat(String userMessage);
}
```

就可以直接用了

### 简单使用

```java
@Autowired
private QwenChatModel model;

@Test
void test2() {
    Assistant assistant = AiServices.create(Assistant.class, model);
    String ans = assistant.chat("你好帅");
    System.out.println(ans);
}
```

> 备注
>
> 在 [Quarkus](https://docs.quarkiverse.io/quarkus-langchain4j/dev/ai-services.html) 和 [Spring Boot](https://docs.langchain4j.info/tutorials/spring-boot-integration#spring-boot-starter-for-declarative-ai-services) 应用程序中， 自动配置会处理创建 `Assistant` bean。 这意味着您不需要调用`AiServices.create(...)`，只需在需要的地方注入/自动装配 `Assistant` 即可。

具体想要实现自动的指定注入方式：

```java
@AiService(wiringMode = EXPLICIT, chatModel = "qwenChatModel")
public interface Assistant {
    String chat(String userMessage);
}
```

注意这里之所以需要指明model是因为项目中注入了多个chatLanguageModel的实现类，无法确定到底用哪个Bean，如果只注入了一个，那就不用了（和@Autowired一样）

```java
@Autowired
private Assistant assistant;

@Test
void test2() {
    String ans = assistant.chat("按时递交纳斯卡极度愤怒");
    System.out.println(ans);
}
```

> **这段代码发生了什么？**
>
> 1. **`@AiService` 是 LangChain4j 提供的 Spring Boot 专用注解**（来自 `langchain4j-spring-boot-starter`）
> 2. 它告诉 Spring：“请为这个接口 **自动生成一个代理实现类**，并注册为 Bean”
> 3. `wiringMode = EXPLICIT` 表示：**手动指定依赖**（而不是自动扫描）
> 4. `chatModel = "qwenChatModel"` 表示：**使用名为 `qwenChatModel` 的 Bean 作为 LLM 后端**
>
> ✅ 所以，当你在其他地方 `@Autowired Assistant assistant;` 时，Spring 注入的其实是一个 **动态生成的代理对象**，它内部调用了 `qwenChatModel`。
>
> **不是你写的类，也不是 LangChain4j 写死的类，而是运行时通过 JDK 动态代理（或 ByteBuddy）生成的实现。**
>
> 流程如下：
>
> ```
> 你的接口 Assistant
>        ↓
> @AiService 注解被 Spring 捕获
>        ↓
> LangChain4j 的 AiServiceBeanPostProcessor 创建代理
>        ↓
> 代理内部持有 qwenChatModel Bean 的引用
>        ↓
> 调用 assistant.chat("xxx") 
>        ↓
> 代理构造 Prompt → 调用 qwenChatModel.generate() → 返回结果
> ```
>
> 💡 这和 MyBatis 的 `@Mapper`、Spring Data JPA 的 `Repository` 原理类似：**接口即实现**。
>
> 说明文档解释：
>
> 您将接口的 `Class` 与低级组件一起提供给 `AiServices`， 然后 `AiServices` 创建一个实现该接口的代理对象。 目前，它使用反射，但我们也在考虑其他替代方案。 这个代理对象处理所有输入和输出的转换。 在这种情况下，输入是单个 `String`，但我们使用的是 `ChatLanguageModel`，它接受 `ChatMessage` 作为输入。 因此，`AiService` 会自动将其转换为 `UserMessage` 并调用 `ChatLanguageModel`。 由于 `chat` 方法的输出类型是 `String`，在 `ChatLanguageModel` 返回 `AiMessage` 后， 它将在从 `chat` 方法返回之前转换为 `String`。



## ChatMemory

之前搞的都是简单的单次对话，怎么实现多次对话记忆实现简单的连贯对话（也即记忆）？

### 上下文连贯记忆

其实封装的model是可以接收List形式的传输的：

```java
default ChatResponse chat(ChatMessage... messages) {
    ChatRequest chatRequest = ChatRequest.builder().messages(messages).build();
    return this.chat(chatRequest);
}

default ChatResponse chat(List<ChatMessage> messages) {
    ChatRequest chatRequest = ChatRequest.builder().messages(messages).build();
    return this.chat(chatRequest);
}
```

但是List形式传输好像还是太low了，有没有别的办法？有的，我们只需要一个额外**记忆组件**即可：

```java
@Autowired
private QwenChatModel model;

@Test
void contextLoads() {
    MessageWindowChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);
    Assistant assistant = AiServices.builder(Assistant.class)
            .chatLanguageModel(model)
            .chatMemory(memory)
            .build();
}
```

这样我们的对话就可以被llm记忆了，可是手动创建还是太low了，但是怎么实现自动管理呢？其实也只需要自己搞一个Bean就行：

```java
@AiService(
        wiringMode = EXPLICIT,
        chatModel = "qwenChatModel",
        chatMemory = "chatMemory"
)
public interface Assistant {
    String chat(String userMessage);
}
```

```java
@Configuration
public class MemoryConfig {

    @Bean
    public ChatMemory chatMemory() {
        return MessageWindowChatMemory.withMaxMessages(10);
    }
}
```

那么这时我们就有个疑问了，Bean一般是**单例模式**，要是多个请求一起用这个Bean，那岂不是共享上下文了，简直乱了套了！

### 会话间上下文隔离

其实就是为每次对话加一个唯一标识ID即可：

```java
@AiService(
        wiringMode = EXPLICIT,
        chatModel = "qwenChatModel",
        chatMemoryProvider = "chatMemoryProvider"
)
public interface Assistant {
    String chat(@MemoryId String memoryId, @UserMessage String userMessage);
}
```

其中：

- **@MemoryId**：标记此字段为对话ID
- **@UserMessage**：标记此字段为用户消息（AI的为AIMessage）

- **chatMemoryProvider**：注入的Bean对象，用于构造对话上下文

```java
@Configuration
public class MemoryConfig {

    @Bean
    public ChatMemoryProvider chatMemoryProvider() {
        return memoryId ->MessageWindowChatMemory.builder().id(memoryId).maxMessages(10).build();
    }
}
```

> **为什么这里可以直接拿到memoryId？**
>
> 在 LangChain4j 中，`ChatMemoryProvider` 是一个 **函数式接口（Functional Interface）**：
>
> ```java
> @FunctionalInterface
> public interface ChatMemoryProvider {
>     ChatMemory get(Object memoryId);
> }
> ```
>
> 也就是说，它只有一个方法：`get(Object memoryId)`，接收一个 `memoryId`，返回一个 `ChatMemory` 实例。
>
> 而你的 Lambda：
>
> ```java
> memoryId -> MessageWindowChatMemory.builder().id(memoryId).maxMessages(10).build()
> ```
>
> 等价于：
>
> ```java
> new ChatMemoryProvider() {
>     @Override
>     public ChatMemory get(Object memoryId) {
>         return MessageWindowChatMemory.builder()
>             .id(memoryId)
>             .maxMessages(10)
>             .build();
>     }
> }
> ```
>
> ✅ 所以 `memoryId` 就是 `get()` 方法的参数！
>
> 当使用Assistant时，自动将标记了@MemoryId的参数传给这个构造器。

然后只管用即可：

```java
@Autowired
private Assistant assistant;

@Test
void contextLoads() {
    String ans = assistant.chat(1,"我是limi");
    System.out.println(ans);
    System.out.println(assistant.chat(1,"我是谁"));
}
```

### 自制会话记忆store

对话多了全存内存里面那就爆了，应用出了问题需要重启，那么过往记忆就会丢失。这时候我们就不得不去持久化了，可是我们要怎么持久化比较好？是在应用层拿到了用户和AI的String字符串再写入外存吗？其实Langchain4j中我们可以直接在他的会话接口中额外加入自己需要的业务操作：

> **`ChatMemoryStore` 是 LangChain4j 用来缓存和管理多个 `ChatMemory` 实例（按 `memoryId`）的存储仓库。**

**它解决的核心问题：**

- 当你有成百上千个用户/工单在对话时，如何：
  - 快速根据 `memoryId` 找到对应的 `ChatMemory`
  - 避免重复创建
  - 控制内存使用（如自动过期）
  - 支持持久化（重启不丢）

具体到设置就是provider中的chatMemoryStore配置：

```java
@Bean
public ChatMemoryProvider chatMemoryProvider() {
    return memoryId ->MessageWindowChatMemory.
            builder().
            id(memoryId).
            maxMessages(10).
            chatMemoryStore(new InMemoryChatMemoryStore()).
            build();
}
```

他的默认就是一个SingleSlotChatMemoryStore（List类型记录对话）：

```java
public Builder chatMemoryStore(ChatMemoryStore store) {
    this.store = store;
    return this;
}

public MessageWindowChatMemory build() {
    return new MessageWindowChatMemory(this);
}

public ChatMemoryStore store() {
    return store != null ? store : new SingleSlotChatMemoryStore(id);
}
```

当然也可以是hashmap的InMemoryChatMemoryStore，其实他们就是实现了一个接口：

```java
public interface ChatMemoryStore {
    List<ChatMessage> getMessages(Object memoryId);
    void updateMessages(Object memoryId, List<ChatMessage> messages);
    void deleteMessages(Object memoryId);
}
```

那么是不是只要我们也去实现这样的一个接口就可以将数据的操作封装到这个实现类中了呢？

> 这样对其他业务代码的侵入性小一点

### 数据库选择

| 方案               | 类型          | 优点                                                         | 缺点                                                         | 制造业适用度                            | LangChain4j 支持                                 | 部署复杂度         | 成本                            |
| ------------------ | ------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | --------------------------------------- | ------------------------------------------------ | ------------------ | ------------------------------- |
| MongoDB            | 文档数据库    | ✅ 原生 JSON 存储 ✅ 灵活 Schema（消息结构可变） ✅ TTL 自动过期 ✅ 高写入吞吐（10w+/s） ✅ 单文档 = 一个会话，天然隔离 | ❌ 不支持 ACID 跨文档事务（但对话通常单文档）                 | ⭐⭐⭐⭐⭐ • 工单级对话 • 操作员问答记录     | ✅ 官方无直接集成，但极易自定义 `ChatMemoryStore` | ⭐⭐ Docker 一行启动 | 💰 开源免费，云托管按量付费      |
| Redis              | 内存数据库    | ✅ 微秒级读写 ✅ 内置 Hash/List 结构 ✅ 支持 TTL ✅ 分布式部署成熟 | ❌ 内存成本高 ❌ 持久化是“兜底”，非主存储 ❌ 大对象（长对话）占内存 | ⭐⭐⭐ • 短期会话缓存 • 实时状态同步       | ✅ 社区有 `RedisChatMemoryStore` 实现             | ⭐ 极简             | 💰💰 内存贵，但对话数据小，可接受 |
| PostgreSQL (JSONB) | 关系型 + JSON | ✅ 强一致性 + 事务 ✅ JSONB 高效查询/索引 ✅ 与 ERP 主库合一，减少系统数 | ❌ 写入性能低于 MongoDB ❌ 需手动管理 JSON 结构版本            | ⭐⭐⭐ • 对一致性要求高的场景 • 小规模部署 | ⚠️ 需自定义实现                                   | ⭐⭐                 | 💰 开源免费                      |

- **MongoDB 最优**：一条对话 = 一个文档，字段 `messages: [ {...}, {...} ]`，完美映射 `List<ChatMessage>`
- **Redis 次之**：可用 Hash 存 `{ "msg_0": "...", "msg_1": "..." }`，但需拼接（但一般就是内存不够用了才需要持久化所以也不会用）
- **PostgreSQL 可用**：用 `JSONB` 字段存消息列表，但更新整块 JSON 效率低

所以我们选择**MongoDB** 



### MongoDB 

MongoDB 是一个开源的、面向文档的 NoSQL 数据库系统，以其高性能、高可用性、可扩展性和灵活的数据模型而广受欢迎。以下是 MongoDB 的核心概念和特点的简要介绍：

- **文档（Document）**：MongoDB 的基本数据单元，类似于 JSON 对象。实际上使用的是 BSON（Binary JSON），支持更多数据类型（如日期、二进制等）。
- **集合（Collection）**：文档的容器，相当于关系型数据库中的“表”，但无需预定义结构（schema-less）。
- **数据库（Database）**：集合的容器，一个 MongoDB 实例可以包含多个数据库。

#### 具体操作

| SQL 操作                                               | MongoDB 操作                                                 | 说明                                                     |
| ------------------------------------------------------ | ------------------------------------------------------------ | -------------------------------------------------------- |
| `SELECT * FROM users`                                  | `db.users.find()`                                            | 查询所有文档                                             |
| `SELECT name, age FROM users`                          | `db.users.find({}, {name: 1, age: 1})`                       | 投影（只返回指定字段）                                   |
| `SELECT * FROM users WHERE age = 25`                   | `db.users.find({age: 25})`                                   | 等值查询                                                 |
| `SELECT * FROM users WHERE age > 25`                   | `db.users.find({age: {$gt: 25}})`                            | 条件查询（ $ gt = greater than）                         |
| `SELECT * FROM users WHERE age >= 25 AND age <= 40`    | `db.users.find({age: {$gte: 25, $lte: 40}})`                 | 多条件在同一字段                                         |
| `SELECT * FROM users WHERE age > 25 AND city = "北京"` | `db.users.find({age: {$gt: 25}, city: "北京"})`              | 多字段 AND（默认就是 AND）                               |
| `SELECT * FROM users WHERE age > 25 OR city = "上海"`  | `db.users.find({$or: [{age: {$gt: 25}}, {city: "上海"}]})`   | OR 查询                                                  |
| `SELECT * FROM users WHERE name LIKE "%三%"`           | `db.users.find({name: /三/})` 或 `db.users.find({name: {$regex: "三"}})` | 模糊匹配（正则）                                         |
| `SELECT * FROM users ORDER BY age DESC`                | `db.users.find().sort({age: -1})`                            | 排序（-1 降序，1 升序）                                  |
| `SELECT * FROM users LIMIT 10`                         | `db.users.find().limit(10)`                                  | 限制返回数量                                             |
| `SELECT * FROM users LIMIT 10 OFFSET 20`               | `db.users.find().skip(20).limit(10)`                         | 分页（跳过前 20 条，取 10 条）                           |
| `INSERT INTO users (name, age) VALUES ('李四', 28)`    | `db.users.insertOne({name: "李四", age: 28})`                | 插入单条文档                                             |
| `UPDATE users SET age = 29 WHERE name = '李四'`        | `db.users.updateOne({name: "李四"}, {$set: {age: 29}})`      | 更新单条（需用 `$set`）                                  |
| `UPDATE users SET age = age + 1 WHERE city = '北京'`   | `db.users.updateMany({city: "北京"}, {$inc: {age: 1}})`      | 批量更新（ $ inc 增量）                                  |
| `DELETE FROM users WHERE age < 18`                     | `db.users.deleteMany({age: {$lt: 18}})`                      | 删除多条文档                                             |
| `DELETE FROM users WHERE name = '王五'`                | `db.users.deleteOne({name: "王五"})`                         | 删除单条                                                 |
| `COUNT(*) FROM users`                                  | `db.users.countDocuments({})`                                | 统计文档总数（推荐用 `countDocuments` 而非旧版 `count`） |
| `SELECT city, COUNT(*) FROM users GROUP BY city`       | 使用聚合管道： `db.users.aggregate([<br>  {$group: {_id: "$city", count: {$sum: 1}}}<br>])` | 分组统计（GROUP BY）                                     |

**常用操作符**

| 操作符        | 含义                       | 示例                                |
| ------------- | -------------------------- | ----------------------------------- |
| `$eq`         | 等于                       | `{age: {$eq: 30}}`                  |
| `$ne`         | 不等于                     | `{age: {$ne: 30}}`                  |
| `$gt`, `$gte` | 大于、大于等于             | `{age: {$gt: 25}}`                  |
| `$lt`, `$lte` | 小于、小于等于             | `{age: {$lte: 40}}`                 |
| `$in`         | 在列表中                   | `{city: {$in: ["北京", "上海"]}}`   |
| `$nin`        | 不在列表中                 | `{city: {$nin: ["深圳"]}}`          |
| `$exists`     | 字段是否存在               | `{email: {$exists: true}}`          |
| `$size`       | 数组长度                   | `{hobbies: {$size: 2}}`             |
| `$elemMatch`  | 数组中至少一个元素满足条件 | `{scores: {$elemMatch: {$gt: 90}}}` |

### SpringBoot整合MongoDB

#### 依赖与配置

依赖和配置方面和mysql差不多，改改驱动，url即可：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-mongodb</artifactId>
</dependency>
```

```yml
spring:
  data:
    mongodb:
      uri: mongodb://localhost:27017/chat_memory_db
```

> 其中这个数据库会自动创建，无需事先准备。在 MongoDB 中，**数据库（database）和集合（collection）通常会在首次写入数据时自动创建**，无需手动预先创建。

#### 使用样例

**实体类：**

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
@Document("chat_messages")
public class ChatMessages {

    // 唯一标识，映射到 MongoDB 文档的 _id 字段
    @Id
    private ObjectId messageId;

    // sessionId，对话id
    private Long memoryId;

    // 存储当前聊天记录列表的json字符串
    private String content; 
}

```

和mybatisplus的大差不差

**工具类：**

```java
@Autowired
private MongoTemplate mongoTemplate;

@Test
void testMongo() {
    ChatMessages chatMessages = new ChatMessages();
    chatMessages.setContent("{\"id\":\"1\",\"role\":\"user\",\"content\":\"我是limi\"}");
    mongoTemplate.insert(chatMessages);

    System.out.println(mongoTemplate.findById("69a682a74397691109ddf2e1", ChatMessages.class));
}
```

### MongoTemplate的操作

准备实体类：

```java
@Document(collection = "users")
public class User {
    @Id
    private String id;
    private String name;
    private int age;
    private List<String> hobbies;
    // 构造函数、getter/setter 略
}
```

> 相比之其他数据库可能还需要orm，MongoDB由于其简单的操作，只需要自己的jdbc就行，指明集合（表）以及主键ID即可。
>
> - @Document(collection = "users")：指明集合
> - @Id：指明主键ID



#### 插入操作（Insert）

```java
// 插入单个
User user = new User("张三", 30, Arrays.asList("读书", "游泳"));
mongoTemplate.insert(user);

// 插入多个
List<User> users = Arrays.asList(
    new User("李四", 25, Arrays.asList("电影")),
    new User("王五", 28, Arrays.asList("游戏", "音乐"))
);
mongoTemplate.insert(users, User.class); // 第二个参数指定集合对应的类
```



#### 查询（Select）

**基础查询（类似 SELECT \* FROM users WHERE ...）**

```java
// 查询所有
List<User> all = mongoTemplate.findAll(User.class);

// 根据 ID 查询
User user = mongoTemplate.findById("65f1a...", User.class);

// 条件查询：age > 25
Query query = new Query(Criteria.where("age").gt(25));
List<User> result = mongoTemplate.find(query, User.class);
```

**复杂条件（AND / OR）**

```java
// age > 25 AND name = "张三"
Query query = new Query(
    Criteria.where("age").gt(25).and("name").is("张三")
);

// age > 25 OR city = "北京"
Query query = new Query(
    new Criteria().orOperator(
        Criteria.where("age").gt(25),
        Criteria.where("city").is("北京")
    )
);
```

**投影（只返回部分字段）**

```java
Query query = new Query(Criteria.where("age").gt(25));
query.fields().include("name").exclude("_id"); // 只返回 name，排除 _id
List<User> users = mongoTemplate.find(query, User.class);
```

> ⚠️ 注意：即使只查部分字段，返回的仍是 `User` 对象，未查的字段为 `null`。

**分页与排序**

```java
Query query = new Query()
    .addCriteria(Criteria.where("age").gt(20))
    .with(Sort.by(Sort.Direction.DESC, "age"))
    .skip(10)   // 跳过前10条
    .limit(5);  // 取5条

List<User> page = mongoTemplate.find(query, User.class);
```



#### 更新（Update）

**更新单条（默认只更新第一个匹配项）**

```java
Query query = new Query(Criteria.where("name").is("张三"));
Update update = new Update().set("age", 31).push("hobbies", "跑步");

mongoTemplate.updateFirst(query, update, User.class);
```

**更新多条**

```java
mongoTemplate.updateMulti(query, update, User.class);
```

**常用 Update 操作符**

| 操作                            | 说明                 |
| :------------------------------ | :------------------- |
| `set("field", value)`           | 设置字段值           |
| `unset("field")`                | 删除字段             |
| `inc("score", 10)`              | 数值字段 +10         |
| `push("tags", "newTag")`        | 向数组末尾添加元素   |
| `addToSet("tags", "uniqueTag")` | 向数组添加不重复元素 |
| `pull("hobbies", "读书")`       | 从数组中移除指定元素 |



#### 删除（Remove）

```java
// 删除单条
Query query = new Query(Criteria.where("name").is("张三"));
mongoTemplate.remove(query, User.class);

// 删除所有匹配
mongoTemplate.remove(new Query(), User.class); // 清空整个集合！慎用
```



#### **聚合操作（Aggregation）**

实现类似 SQL 的 `GROUP BY`、`JOIN`（ $ lookup）、复杂计算等。

**示例：按年龄分组统计人数**

```java
Aggregation aggregation = Aggregation.newAggregation(
    Aggregation.match(Criteria.where("age").gt(20)),
```

#### 条件构造详解

之前用过mybatisplus的querywrapper的话，其实MongoDB的大差不差，来看看对比：

| 功能             | MyBatis-Plus (`QueryWrapper`)                  | Spring Data MongoDB (`Criteria`)                             |
| :--------------- | :--------------------------------------------- | :----------------------------------------------------------- |
| **所属框架**     | MyBatis-Plus（ORM for SQL）                    | Spring Data MongoDB（ODM for NoSQL）                         |
| **底层语言**     | 生成 SQL 语句（如 `WHERE name = '张三'`）      | 构建 BSON 查询对象（如 `{name: "张三"}`）                    |
| **核心类**       | `QueryWrapper<T>`                              | `Criteria` + `Query`                                         |
| **链式调用**     | ✅ 支持（如 `.eq().like()`）                    | ✅ 支持（如 `.where("age").gt(25)`）                          |
| **字段引用方式** | 通常用字符串（或 Lambda 表达式）               | **必须用字符串字段名**（因 MongoDB 无固定 schema）           |
| **AND / OR**     | `.and(i -> i.eq(...))` / `.or(i -> i.eq(...))` | `Criteria.where(...).and(...)`（默认 AND） `new Criteria().orOperator(c1, c2)` |
| **嵌套条件**     | 支持通过 lambda 嵌套                           | 需手动组合多个 `Criteria` 对象                               |

常用条件构造：

| 查询需求         | MyBatis-Plus (`QueryWrapper`)                                | Spring Data MongoDB (`Criteria`)                             |
| :--------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **等于**         | `.eq("name", "张三")`                                        | `Criteria.where("name").is("张三")`                          |
| **不等于**       | `.ne("age", 25)`                                             | `Criteria.where("age").ne(25)`                               |
| **大于**         | `.gt("age", 25)`                                             | `Criteria.where("age").gt(25)`                               |
| **小于等于**     | `.le("score", 100)`                                          | `Criteria.where("score").lte(100)`                           |
| **IN**           | `.in("city", Arrays.asList("北京", "上海"))`                 | `Criteria.where("city").in(Arrays.asList("北京", "上海"))`   |
| **NOT IN**       | `.notIn("status", "deleted")`                                | `Criteria.where("status").nin("deleted")`                    |
| **LIKE（模糊）** | `.like("name", "三")` → `%三%`                               | `Criteria.where("name").regex(".*三.*", "i")` 或 `Criteria.where("name").regex(Pattern.compile("三"))` |
| **IS NULL**      | `.isNull("email")`                                           | `Criteria.where("email").is(null)` 或 `Criteria.where("email").exists(false)` |
| **IS NOT NULL**  | `.isNotNull("email")`                                        | `Criteria.where("email").exists(true)`                       |
| **BETWEEN**      | `.between("age", 20, 30)`                                    | `Criteria.where("age").gte(20).lte(30)`                      |
| **多条件 AND**   | `.eq("a", 1).eq("b", 2)`                                     | `Criteria.where("a").is(1).and("b").is(2)`                   |
| **多条件 OR**    | `.or().eq("a", 1).eq("b", 2)`                                | `new Criteria().orOperator(<br> Criteria.where("a").is(1),<br> Criteria.where("b").is(2)<br>)` |
| **排序**         | `.orderByDesc("createTime")`                                 | `new Query().with(Sort.by(Sort.Direction.DESC, "createTime"))` |
| **分页**         | `Page<T> page = new Page<>(1, 10);` `baseMapper.selectPage(page, wrapper);` | `query.skip(0).limit(10)`                                    |



### 利用MongoDB实现记忆store的封装

在完成了以上的学习之后，我们就可以将会话存在自己配置的外存中了，只需要将会话内容存成json字符串，将上下文的隔离ID额外再做一层封装即可完成数据的实现：

首先我们先来实现一个自制的store：

```java
@Component
public class MyMemoryStore implements ChatMemoryStore {

    // 只取持久化数据，无缓存

    @Autowired
    private MongoTemplate mongoTemplate;

    /**
     * 获取缓存
     * @param memoryId 缓存id
     * @return 缓存内容
     */
    @Override
    public List<ChatMessage> getMessages(Object memoryId) {
        Criteria criteria = Criteria.where("memoryId").is(memoryId);
        Query query = new Query(criteria);
        ChatMessages chatMessages = mongoTemplate.findOne(query, ChatMessages.class);
        if (chatMessages == null) {
            return new LinkedList<>();
        }
        String content = chatMessages.getContent();
        return ChatMessageDeserializer.messagesFromJson(content);
    }

    /**
     * 更新缓存
     * @param memoryId 消息id
     * @param messages 消息内容
     */
    @Override
    public void updateMessages(Object memoryId, List<ChatMessage> messages) {
        Criteria criteria = Criteria.where("memoryId").is(memoryId);
        Query query = new Query(criteria);

        Update update = new Update();
        update.set("content", ChatMessageSerializer.messagesToJson(messages));

        mongoTemplate.upsert(query, update, ChatMessages.class);
    }

    /**
     * 删除缓存
     * @param memoryId 缓存id
     */
    @Override
    public void deleteMessages(Object memoryId) {
        Criteria criteria = Criteria.where("memoryId").is(memoryId);
        Query query = new Query(criteria);
        mongoTemplate.remove(query, ChatMessages.class);
    }
}
```

然后我们将之注入到chatmemory配置中：

```java
@Configuration
public class MemoryConfig {

    @Autowired
    private MyMemoryStore myMemoryStore;

    @Bean
    public ChatMemoryProvider chatMemoryProvider() {
        return memoryId ->MessageWindowChatMemory.
                builder().
                id(memoryId).
                maxMessages(10).
                chatMemoryStore(myMemoryStore).
                build();
    }
}
```

确保注入到了assistant接口中：

```java
@AiService(
        wiringMode = EXPLICIT,
        chatModel = "qwenChatModel",
        chatMemoryProvider = "chatMemoryProvider"
)
public interface Assistant {
    String chat(@MemoryId Integer memoryId, @UserMessage String userMessage);
}
```

然后就可以只管用了。

### 流程总结

其实就是每次调用llm回答的时候需要根据**上下文隔离ID**获取到对话记录保存到当前的chatmemory中，关键是需要按照Langchain4j给出的规划来，也即实现**ChatMemoryStore**，并且存储/取出的结果得是**符合要求**的对话记录（调用**ChatMessageDeserializer**进行序列化/反序列化），至于数据怎么存储，从哪里来，是内存还是外存，Langchain4j都不管。

- 后端根据用户环境生成/拿到上下文隔离ID，然后将消息String交给Langchain4j进行llm调用
- Langchain4j拿到上下文隔离ID，根据ID从**ChatMemoryStore**取出过往记忆和当前的消息一拼得到**最新对话记录ChatMemory**
- 拿着最新对话记录ChatMemory去调用llm，得到llm响应内容String，存入最新对话记录ChatMemory
- 将最新对话记录ChatMemory存入ChatMemoryStore（通过提供的接口实现）

就是对话记录json不停拼来拼去，没有特别复杂的调用。
