缓存

SaltFishGC2025-1-7

Redis缓存

查询循序：

​ 前端先查缓存->缓存存在：返回数据->缓存不存在该数据：查询数据库

​ 查询数据库->存在/不存在：返回结果，若存在则更新缓存

好处：

读写快，响应速度快，降低数据库压力

坏处：

数据一致性要求（缓存和数据库数据要求一致），运维成本

符合缓存的数据要求：

  1. 数据量没有大到离谱
  2. 对该数据QPS较高

设置缓存尽量加上ttl

缓存更新策略

redis毕竟还是吃内存的，假如数据量达到设置的阈值了，再加数据怎么办？

内存淘汰

redis自带，快到上限了就会自动删除部分数据以空出一部分空间，用户下次查不到了再加

超时删除

常见的ttl，过期自动删除

主动更新

自主设计修改数据请求

主动更新策略

上面我们说了存储上限到了，用新数据来替换旧数据的策略，这里我们再讲讲将一个缓存数据更新该怎么操作？

  1. 同时：更新数据库的同时更新缓存**（最常使用）**
  2. 写穿：数据库与缓存高内聚，只管用
  3. 写回：只管更新缓存，缓存通过rdb快照，aof等方式异步持久化写回数据库（redis全歇菜了就完蛋，需要集群哨兵等保证缓存可靠性）

高一致性需求：分布式锁（非常需要数据一致/数据安全时）

低一致性需求：先读再说

当然以上都是对存入缓存的数据讨论的，对于qps不高的数据，直接读数据库甚至HBase的慢读都可以。

数据更新后对缓存的处理

这里还需要注意一下数据库数据更新完了，我们是直接更新缓存数据，还是先删除了，等以后查询了再将新数据加到缓存里面好：

直接更新：假如更新多，查询少，那就会导致每次更新还得再改一下缓存

直接删了：更新完了没有这个缓存，下次查询完了再加入缓存，多次更新间无需执行修改缓存操作**（推荐）**

更新操作中，数据库以及缓存的处理顺序

其实都可以：

先改数据库后删缓存：删缓存期间会脏读，同时也可能会导致旧数据写缓存，但相对概率小**（推荐）**

先删缓存后改数据库：改数据库未提交期间其他事务读了旧数据可能会将旧数据写到缓存中，需要延迟双删

低数据一致性：延迟双删（一定要确保延迟时间大于一个请求的处理时间）

高数据一致性：分布式锁（要确保阻塞带来的影响不会过大）

用户更新数据导致数据库以及缓存不一致的问题以及其所带来的脏读问题，怎么解决？

延迟双删：彻底更新完了再删除一次redis中的缓存

分布式锁：直接加锁，拿不到锁不准查，直接阻塞住

对qps以及更新频率有一定要求，高qps下多次更新所带来的延迟非常破坏用户体验

简单总结缓存更新问题

  1. 低一致性需求： 使用 Redis 自带的内存淘汰机制（如 LRU、LFU 等），允许缓存数据过期或被驱逐，适用于对数据实时性要求不高的场景。
  2. 高一致性需求： 采用主动更新（先改数据库后删缓存，延迟双删/分布式锁）策略，并以超时剔除作为兜底方案，确保缓存与数据库的数据一致性。

◆ 读操作：

  • 缓存命中 → 直接返回结果（此步考虑添加刷新超时时间）；
  • 缓存未命中 → 查询数据库，将结果写入缓存，并设定合理的超时时间。

◆ 写操作：

  • 先写数据库，然后再删除缓存（避免脏读）；
  • 要确保数据库写入与缓存删除操作的原子性（可通过事务、消息队列或延迟双删甚至分布式锁等机制实现）。

缓存穿透

问题定义：查询的数据在缓存以及数据库都均不存在，但是始终有请求查询且数量不少，导致每次都查询数据库浪费资源，缓存对于这些数据的查询失效了，称为缓存穿透。

解决方案：

1. 缓存空数据

数据库查不到就缓存一个空对象（只有一个key，对应value为空）顺带上一个短期ttl，当缓存查得数据为空时表示不存在该数据

但是这样的记录方式会占用一些额外内存

且插入新数据时需要检查是否存在该缓存，不处理空缓存会导致短期查询查不到新插入的数据（当然也可以直接删了对应缓存key了事）

当然修改本来就要删除缓存所以不会产生影响

2. 布隆过滤器

简单理解为使用byte数组存储数据，利用hash算法计算hash值，以hashcode的方式存储key。当然这里面也会存在hash算法常见的冲突问题（不同的key得到最终同样的hashcode或者最终索引一致），简单使用中不对这一方面做处理就会导致不准确的问题，即布隆过滤器返回的**“存在”结果可能有误**（A与B不同，但是hash的结果一致，可是A存在于缓存，但是B不存在，向布隆过滤器输入B会发现结果存在，但是事实上没有，但是不存在一定是没有的）

缓存雪崩

**问题定义：**同一时间大量缓存key失效（如redis服务宕机，当然可以通过设置集群哨兵等方案尽可能避免这种情况的发生），请求全部打到数据库，导致数据库压力过大导致崩溃。

解决方案：

  1. 给不同的key加随机的ttl：防止同一时间大量缓存失效。
  2. 强化redis的可靠性：上集群，哨兵（推荐）
  3. 给缓存加限流
  4. 给业务加多级缓存：nginx加缓存（但是数据一致性等方面较难处理，简单实际生产环境不推荐，当然大型项目做好规划就可以了）

缓存击穿

**问题定义：**一个被高并发访问的key突然失效导致大量请求打到数据库，这就会导致短时间（key尚未被更新到redis中前）数据库压力骤升

解决方案：

  1. **互斥锁：**每个进程在查询redis失败时，会尝试获取一个锁以查询数据库并缓存在redis，最后释放，如果有其他进程先行一步获取到了锁，就会阻塞自己（retry多次直到锁释放/超时）
  2. **逻辑过期：**发生此类问题的常见业务场景是ttl超时（查询到了该缓存但不会刷新该缓存过期时间），那么我们可以考虑使用逻辑过期，不设置ttl，但是在每个缓存value自己记录时间戳，过期时间等。当发现缓存过期了，就会开一个新的线程，让它获取一个锁，随后让它将数据库对应信息更新到缓存中并释放锁，而自己直接返回原来的结果。这其中倘若获取锁失败，就会直接返回旧的结果。（但其实更新，删除操作也会同步修改缓存，甚至会加锁保证一致性，其实个人感觉这里更需要考虑ttl刷新问题，以这个方式处理缓存击穿有点掩耳盗铃）而且这种形式存储起来缓存其实就算是持久化了，和数据库大差不差，还要考虑插入新数据，修改时怎么操作缓存的问题

利用redis实现锁-setnx

之前想过使用redis来做一个互斥锁，毕竟redis是单线程的，A先来，A执行完了才会执行B。但当初想的是：

if(redisUtil.getLock(lockId)==null){
	redisUtil.setLock(lockId);
    ...
}

可以看到这里对锁的获取以及上锁是两条命令，这就会出现问题了，A在检测时发现没有，然后准备上锁，但是此时B刚好也检测了发现没有，就会导致一块执行相应操作的问题！

那么redis有没有提供一个命令能够实现尝试获取锁和上锁在一次执行时同时进行呢？有的xd，有的。那就是setnx key value

setnx会在只有没有这个key值时设置一个string类型的键值对，并返回value值，假如已经存在这个key了，就会返回0。我们就可以基于这一条命令来保证锁的准确性，保证同一时间只能有一个进程能获取到锁以及执行相应操作，并精准阻塞其他进程。

注意：setnx上锁时不要设置value为0，会导致冲突

新开线程以异步实现所需功能

我们知道java中创建一个线程可以new Thread().start()这样来实现，但是这种方式需要频繁创建销毁线程，且资源不可控，仅可用于简易场景比如学的时候用，属于下下策。那么我们就考虑使用可控资源来实现这个需求：

  1. 使用 ExecutorService（灵活控制线程池）
// 创建一个固定大小的线程池（推荐用 ThreadPoolTaskExecutor 在 Spring 中）
ExecutorService executor = Executors.newFixedThreadPool(5);

// 提交任务
executor.submit(() -> {
    // 你的业务逻辑
    System.out.println("我在新线程中运行: " + Thread.currentThread().getName());
});

// 应用关闭时优雅关闭（重要！）
executor.shutdown();

默认状态下，ExecutorService 提交任务本身是“同步”的（立即返回），但任务的执行是“异步”的（在后台线程运行）。

操作是同步还是异步？说明
executor.submit(...) 或 execute(...) 调用✅ 同步（立即返回）主线程不会等待任务执行完，而是立刻继续往下走
提交的任务（Runnable/Callable 内部逻辑）✅ 异步（在后台线程执行）实际业务逻辑在另一个线程中运行

那么默认状态下，如果你需要去获取新开线程的执行结果，也就是想要阻塞等待返回结果的话：

Future<String> future = executor.submit(() -> {
    // 模拟查询数据库
    Thread.sleep(1000);
    return "查询结果: 用户ID=123";
});

// 主线程可以继续做其他事...
System.out.println("主线程继续执行...");

// 需要结果时再 get（会阻塞直到完成）
try {
    String result = future.get(); // ← 阻塞等待
    System.out.println(result);
} catch (Exception e) {
    e.printStackTrace();
}

我们可以简单把Future理解为js中的Promise，他就是将执行的过程封装为这个东西，可用于获取最终的执行结果（未结束时尝试获取会进行阻塞）

  1. 使用CompletableFuture（链式异步编程）
CompletableFuture<Void> future = CompletableFuture.runAsync(() -> {
    // 异步发短信
    smsService.sendSms(phone, code);
}, executor); // 可指定线程池

// 可继续链式操作
future.thenRun(() -> {
    log.info("短信发送完成");
});
  1. 使用 @Async 另建一个Service

启用异步支持：

@SpringBootApplication
@EnableAsync // 👈 关键注解
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

定义异步方法：

@Service
public class SmsService {

    @Async // 👈 标记为异步
    public void sendCodeAsync(String phone, String code) {
        // 模拟发短信（耗时 2 秒）
        try { Thread.sleep(2000); } catch (InterruptedException e) {}
        System.out.println("短信已发送: " + phone + " -> " + code);
    }
}

调用（无需关心线程）：

@RestController
public class UserController {

    @Autowired
    private SmsService smsService;

    @PostMapping("/send")
    public String send(@RequestParam String phone) {
        String code = "123456";
        smsService.sendCodeAsync(phone, code); // 立即返回，不等待
        return "验证码已发送，请注意查收";
    }
}
